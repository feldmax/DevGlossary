## Краткий список терминов
- **Algorithm** — конечная последовательность шагов для решения задачи.
- **Data Structure** — способ организации и хранения данных для эффективного доступа и изменения.
- **Array** — последовательность элементов в непрерывной области памяти.
- **Linked List** — последовательность узлов, связанных ссылками друг с другом.
- **Stack** — структура данных по принципу LIFO.
- **Queue** — структура данных по принципу FIFO.
- **Deque** — двусторонняя очередь с операциями на обоих концах.
- **Hash Table** — структура данных для быстрого доступа по ключу через хеш-функцию.
- **Tree** — иерархическая структура из узлов и связей между родителями и потомками.
- **Graph** — структура из вершин и ребер для описания произвольных связей.
- **Heap** — структура данных для быстрого доступа к минимальному или максимальному элементу.
- **Trie** — префиксное дерево для хранения строк или последовательностей.
- **Big O** — обозначение асимптотической оценки роста стоимости алгоритма.
- **Time Complexity** — зависимость времени работы алгоритма от размера входа.
- **Space Complexity** — зависимость объема используемой памяти от размера входа.
- **Amortized Complexity** — средняя стоимость операции на длинной последовательности операций.
- **Recursion** — решение задачи через вызов функции самой себя.
- **Iteration** — решение задачи через циклическое повторение шагов.
- **Memoization** — кеширование результатов вычислений для повторного использования.
- **Dynamic Programming** — метод решения задачи через переиспользование подзадач.
- **Greedy Algorithm** — алгоритм, который на каждом шаге выбирает локально лучший вариант.

## Algorithm
**Кратко:**
Algorithm — это точная, конечная и воспроизводимая последовательность действий, которая переводит входные данные в нужный результат.

**Полное определение:**
Алгоритм — это формализованный способ решения задачи. Он должен быть достаточно однозначным, чтобы его можно было реализовать программно, и достаточно конечным, чтобы выполнение завершалось за разумное число шагов. Один и тот же результат иногда можно получить разными алгоритмами, но с разной стоимостью по времени, памяти и сложности реализации.

**Назначение / зачем это нужно:**
Алгоритмы нужны, чтобы системно решать задачи, сравнивать подходы и выбирать реализацию не по интуиции, а по свойствам: корректности, сложности, устойчивости и простоте сопровождения.

**Когда применять:**
Всегда, когда задача требует воспроизводимого способа обработки данных: поиск, сортировка, маршрутизация, агрегация, планирование, парсинг.

**Ограничения / частые ошибки:**
Частая ошибка — путать идею алгоритма с конкретной реализацией. Еще одна ошибка — выбирать «теоретически быстрый» алгоритм, не учитывая размер данных, константные факторы и простоту кода.

**Примеры:**
- Бинарный поиск в отсортированном массиве.
- Алгоритм Дейкстры для поиска кратчайшего пути.

**Связанные термины:**
Data Structure, Big O, Time Complexity, Correctness

## Data Structure
**Кратко:**
Data structure — это способ организовать данные так, чтобы нужные операции над ними были удобными и эффективными.

**Полное определение:**
Структура данных определяет, как значения хранятся, как между ними выражаются связи и какие операции можно выполнять эффективно: доступ, вставка, удаление, поиск, перебор, приоритизация. Выбор структуры данных напрямую влияет на свойства алгоритма.

**Назначение / зачем это нужно:**
Она нужна, чтобы адаптировать хранение данных под характер операций. Нет универсально лучшей структуры: хорошая структура определяется задачей.

**Когда применять:**
Когда нужно проектировать модель данных в памяти, выбирать контейнер, строить индексы, очереди, графы, кеши, буферы, деревья поиска.

**Ограничения / частые ошибки:**
Частая ошибка — использовать первую попавшуюся структуру без анализа профиля операций. Например, хранить все в списке, хотя критичен быстрый поиск по ключу.

**Примеры:**
- `Dictionary` или `map` для быстрого доступа по ключу.
- Очередь задач для worker pool.

**Связанные термины:**
Algorithm, Array, Hash Table, Tree, Graph

## Array
**Кратко:**
Array — это последовательность элементов, расположенных подряд в памяти, обычно с быстрым доступом по индексу.

**Полное определение:**
Массив хранит элементы одинакового типа в непрерывной области памяти. Благодаря этому элемент по индексу обычно находится за O(1). Но вставка и удаление в середине массива часто требуют сдвига элементов.

**Назначение / зачем это нужно:**
Массив удобен, когда важны компактное хранение, быстрый индексный доступ и предсказуемое поведение в памяти.

**Когда применять:**
Когда данные часто читаются по индексу, когда нужен эффективный перебор и когда число вставок в середину невелико.

**Ограничения / частые ошибки:**
Не стоит считать массив хорошим выбором для частых вставок и удалений в начале или середине. Динамические массивы еще и периодически расширяются с перераспределением памяти.

**Примеры:**
- Список значений, к которым часто обращаются по позиции.
- Буфер для обработки числовых данных.

**Связанные термины:**
Time Complexity, Linked List, Iteration, Cache Locality

## Linked List
**Кратко:**
Linked list — это последовательность узлов, где каждый узел хранит значение и ссылку на следующий, а иногда и на предыдущий узел.

**Полное определение:**
Связный список не требует непрерывной памяти для всех элементов сразу. Он удобен для локальных вставок и удалений, если уже есть ссылка на нужное место. Но произвольный доступ по индексу обычно медленный, потому что требует прохода по узлам.

**Назначение / зачем это нужно:**
Полезен там, где структура часто меняется по связям, а не по индексам, и где важны дешевые операции вставки/удаления около известных узлов.

**Когда применять:**
Когда известна позиция вставки или удаления и нет требования к быстрому доступу по индексу.

**Ограничения / частые ошибки:**
На практике список часто проигрывает массиву из-за плохой locality of reference и накладных расходов на указатели. Новички часто переоценивают его полезность.

**Примеры:**
- Реализация LRU-списка вместе с hash table.
- Двусвязный список для очереди с быстрым удалением узла.

**Связанные термины:**
Array, Pointer, Reference, Deque

## Stack
**Кратко:**
Stack — это структура данных, где последний добавленный элемент извлекается первым.

**Полное определение:**
Стек реализует дисциплину LIFO: push добавляет элемент наверх, pop снимает верхний элемент. Эта модель естественно соответствует вложенности вызовов, откату шагов и парсингу выражений.

**Назначение / зачем это нужно:**
Стек нужен для управления контекстом, обработки вложенных структур, backtracking и организации вычислений с последним активным состоянием.

**Когда применять:**
Когда требуется возврат к последнему состоянию, обход дерева в глубину, вычисление выражений, обработка скобок, undo.

**Ограничения / частые ошибки:**
Стек не подходит, когда нужен доступ не только к верхнему элементу. Также важно отличать структуру данных stack от памяти call stack runtime.

**Примеры:**
- Стек вызовов функций.
- Алгоритм DFS с явным стеком.

**Связанные термины:**
Queue, Recursion, Iteration, DFS

## Queue
**Кратко:**
Queue — это структура данных, где элементы извлекаются в порядке поступления.

**Полное определение:**
Очередь реализует дисциплину FIFO: новые элементы добавляются в хвост, а извлекаются из головы. Это отражает естественную модель обработки задач по порядку поступления.

**Назначение / зачем это нужно:**
Очередь нужна для буферизации, планирования, согласования producer/consumer и упорядоченной обработки событий.

**Когда применять:**
Когда нужно обрабатывать элементы по времени поступления: задания, сообщения, события, сетевые пакеты.

**Ограничения / частые ошибки:**
Обычный список или массив с удалением из начала может быть неэффективным. Для очереди лучше использовать кольцевой буфер, deque или специализированную реализацию.

**Примеры:**
- Очередь задач в background worker.
- Очередь входящих сообщений из сокета.

**Связанные термины:**
Deque, Backpressure, Producer-Consumer, Buffer

## Deque
**Кратко:**
Deque — это очередь, которая поддерживает эффективные операции добавления и удаления с обоих концов.

**Полное определение:**
Deque, или double-ended queue, объединяет свойства очереди и частично стека. Она позволяет работать и с головой, и с хвостом коллекции без дорогих перестроений.

**Назначение / зачем это нужно:**
Deque удобен, когда направление работы зависит от сценария: sliding window, двусторонняя обработка, реализация очередей и стеков на одной структуре.

**Когда применять:**
Когда требуется доступ к обоим концам структуры, например для оконных алгоритмов, кешей или очередей задач с приоритетом по краям.

**Ограничения / частые ошибки:**
Не стоит использовать deque как замену произвольному списку с быстрым доступом в середину — это не его сильная сторона.

**Примеры:**
- Sliding window maximum.
- Реализация очереди и стека одним контейнером.

**Связанные термины:**
Queue, Stack, Linked List, Buffer

## Hash Table
**Кратко:**
Hash table — это структура данных, которая хранит пары ключ-значение и обеспечивает быстрый доступ по ключу в среднем случае.

**Полное определение:**
Хеш-таблица вычисляет для ключа хеш и использует его, чтобы определить место хранения значения. При хорошем распределении ключей операции поиска, вставки и удаления обычно имеют среднюю сложность O(1), хотя в худшем случае возможна деградация.

**Назначение / зачем это нужно:**
Она нужна для быстрого адресного доступа по ключу: словари, индексы, кеши, множества, таблицы сопоставления.

**Когда применять:**
Когда важны быстрые операции lookup по уникальному ключу и не требуется естественный упорядоченный обход элементов.

**Ограничения / частые ошибки:**
Порядок элементов часто не гарантируется. Также важно корректно реализовывать хеширование и сравнение ключей. Коллизии и плохой load factor ухудшают производительность.

**Примеры:**
- `Dictionary<TKey, TValue>` или `map[string]Value`.
- Множество уникальных идентификаторов пользователей.

**Связанные термины:**
Key, Collision, Load Factor, Set, Cache

## Tree
**Кратко:**
Tree — это иерархическая структура данных, в которой каждый узел связан с потомками и обычно имеет одного родителя.

**Полное определение:**
Деревья используются для представления вложенности, упорядочивания и эффективного поиска. Существуют разные виды деревьев: бинарные, сбалансированные, B-tree, trie, AST. Свойства конкретного дерева определяют эффективность операций.

**Назначение / зачем это нужно:**
Дерево удобно для моделирования иерархий, индексации, поиска, разбора выражений, файловых систем и структур документов.

**Когда применять:**
Когда данные естественно иерархичны или когда нужен упорядоченный поиск и диапазонные запросы.

**Ограничения / частые ошибки:**
Несбалансированное дерево может деградировать по сложности. Важно выбирать тип дерева под конкретный профиль операций.

**Примеры:**
- AST компилятора.
- Индекс базы данных на B-tree.

**Связанные термины:**
Graph, Trie, Heap, Balanced Tree, DFS

## Graph
**Кратко:**
Graph — это структура данных, описывающая произвольные связи между сущностями через вершины и ребра.

**Полное определение:**
Граф обобщает дерево: в нем могут быть циклы, несколько путей между узлами, направленные и ненаправленные связи, веса ребер. Он подходит для задач, где важна не иерархия, а сеть отношений.

**Назначение / зачем это нужно:**
Графы нужны для маршрутизации, зависимостей, социальных связей, workflow, рекомендаций, анализа сетей и состояний.

**Когда применять:**
Когда сущности образуют сложную сеть связей, а не простую последовательность или дерево.

**Ограничения / частые ошибки:**
Частая ошибка — пытаться представить граф как набор списков или таблиц без явной модели связей, из-за чего алгоритмы усложняются. Для больших графов важен выбор представления: список смежности или матрица.

**Примеры:**
- Граф зависимостей пакетов.
- Карта дорог с весами на ребрах.

**Связанные термины:**
Tree, Vertex, Edge, BFS, DFS

## Heap
**Кратко:**
Heap — это структура данных, поддерживающая быстрый доступ к минимальному или максимальному элементу.

**Полное определение:**
Куча обычно реализуется как почти полное бинарное дерево с инвариантом: родитель не больше детей в min-heap или не меньше в max-heap. Благодаря этому верхний элемент всегда экстремален, а операции вставки и извлечения работают за O(log n).

**Назначение / зачем это нужно:**
Heap нужен, когда важно часто извлекать элемент с наивысшим или наименьшим приоритетом.

**Когда применять:**
В приоритетных очередях, планировщиках, алгоритмах графов, top-k задачах, потоковой обработке.

**Ограничения / частые ошибки:**
Куча не дает быстрого поиска произвольного элемента и не хранит элементы полностью отсортированными. Это не то же самое, что memory heap.

**Примеры:**
- Планировщик задач по приоритету.
- Алгоритм Дейкстры с priority queue.

**Связанные термины:**
Priority Queue, Tree, Big O, Scheduling

## Trie
**Кратко:**
Trie — это дерево, в котором путь от корня кодирует префикс строки или последовательности.

**Полное определение:**
Префиксное дерево хранит ключи посимвольно или поэлементно. Общие префиксы не дублируются, поэтому структура удобна для операций автодополнения, поиска по префиксу и словарных проверок.

**Назначение / зачем это нужно:**
Trie нужен, когда ключевой операцией является поиск по префиксу, а не только точное совпадение.

**Когда применять:**
Для autocomplete, поиска слов, маршрутизации по сегментам, хранения набора строк с общими началами.

**Ограничения / частые ошибки:**
Trie может потреблять много памяти, особенно при редких или длинных ключах и плохой компрессии узлов.

**Примеры:**
- Поиск подсказок в строке ввода.
- Проверка, начинается ли строка с допустимого маршрута.

**Связанные термины:**
Tree, Prefix Search, Dictionary, String Matching

## Big O
**Кратко:**
Big O — это способ описать, как растет стоимость алгоритма при увеличении размера входных данных.

**Полное определение:**
Обозначение Big O скрывает константы и менее значимые члены, чтобы показать порядок роста: O(1), O(log n), O(n), O(n log n), O(n²) и так далее. Это инструмент для сравнения масштабируемости алгоритмов, а не точного измерения времени.

**Назначение / зачем это нужно:**
Big O помогает оценивать поведение алгоритма на больших входах и объяснять, почему один подход перестанет быть практичным при росте данных.

**Когда применять:**
При проектировании алгоритмов, выборе структуры данных и сравнении вариантов реализации.

**Ограничения / частые ошибки:**
Big O не показывает реальные миллисекунды, не учитывает константы, устройство памяти, кеши CPU, особенности данных и среды выполнения. Его нельзя использовать в отрыве от практики.

**Примеры:**
- Линейный поиск в массиве — O(n).
- Бинарный поиск — O(log n).

**Связанные термины:**
Time Complexity, Space Complexity, Amortized Complexity, Scalability

## Time Complexity
**Кратко:**
Time complexity — это оценка того, как число шагов алгоритма растет вместе с размером входа.

**Полное определение:**
Временная сложность описывает асимптотическую стоимость выполнения. Обычно считают количество элементарных операций в зависимости от `n`. Это помогает понять, как алгоритм масштабируется и где у него потенциальные узкие места.

**Назначение / зачем это нужно:**
Она нужна для выбора алгоритмов, особенно когда данные могут расти и текущая реализация перестанет укладываться в допустимое время.

**Когда применять:**
При анализе производительности, код-ревью, проектировании API и инфраструктуры обработки данных.

**Ограничения / частые ошибки:**
Нельзя путать асимптотическую оценку с профилированием. Иногда O(n) с хорошими константами лучше, чем O(log n) на маленьких данных и сложной структуре.

**Примеры:**
- Один цикл по массиву — обычно O(n).
- Два вложенных цикла по одному массиву — часто O(n²).

**Связанные термины:**
Big O, Space Complexity, Profiling, Hot Path

---

## Space Complexity
**Кратко:**
Space complexity — это оценка того, как объем используемой памяти зависит от размера входа.

**Полное определение:**
Пространственная сложность отражает не только хранение входных данных, но и дополнительную память: буферы, таблицы, стек вызовов, вспомогательные структуры. Иногда алгоритм быстрее, но дороже по памяти, и наоборот.

**Назначение / зачем это нужно:**
Она помогает контролировать потребление памяти, что особенно важно для больших данных, embedded-систем, high-load сервисов и алгоритмов с большим числом промежуточных структур.

**Когда применять:**
Когда память ограничена или когда нужно сравнить in-place решение с решением через дополнительный буфер.

**Ограничения / частые ошибки:**
Частая ошибка — анализировать только время и игнорировать то, что алгоритм создает слишком много временных объектов или таблиц.

**Примеры:**
- Сортировка in-place требует меньше дополнительной памяти, чем копирующая сортировка.
- Memoization ускоряет вычисления ценой дополнительной памяти.

**Связанные термины:**
Big O, Time Complexity, Heap, Stack

---

## Amortized Complexity
**Кратко:**
Amortized complexity — это средняя стоимость операции на длинной последовательности операций, даже если отдельные из них иногда дороги.

**Полное определение:**
Некоторые структуры данных выполняют большинство операций дешево, но время от времени делают дорогую перестройку. Амортизированный анализ распределяет стоимость редких дорогих операций на множество дешевых, чтобы показать реальную среднюю цену использования структуры.

**Назначение / зачем это нужно:**
Он нужен для корректной оценки контейнеров и алгоритмов, где изредка происходят resize, rehash, compaction или иные перестройки.

**Когда применять:**
При анализе динамических массивов, хеш-таблиц, persistent-структур и других систем с редкими дорогими фазами.

**Ограничения / частые ошибки:**
Амортизированная O(1) не означает, что каждая операция O(1). В latency-sensitive системах редкие дорогие всплески все равно могут быть проблемой.

**Примеры:**
- Добавление в dynamic array обычно амортизированно O(1), хотя resize дорогой.
- Rehash в hash table редок, но затратен.

**Связанные термины:**
Big O, Dynamic Array, Hash Table, Latency

## Recursion
**Кратко:**
Recursion — это подход, при котором функция решает задачу через вызов самой себя на более простой версии той же задачи.

**Полное определение:**
Рекурсивное решение обычно состоит из базового случая и рекурсивного перехода. Это естественный способ работать с деревьями, графами, divide-and-conquer и формально рекурсивными определениями.

**Назначение / зачем это нужно:**
Рекурсия помогает выразить алгоритм ближе к структуре самой задачи и иногда делает код короче и понятнее.

**Когда применять:**
Для обходов деревьев, DFS, разбиения задачи на подзадачи, вычисления выражений, парсинга.

**Ограничения / частые ошибки:**
Глубокая рекурсия может привести к переполнению стека. Также рекурсивный код бывает медленнее или менее прозрачен для отладки, чем итеративный.

**Примеры:**
- Обход бинарного дерева.
- Merge sort и quicksort.

**Связанные термины:**
Iteration, Stack, Dynamic Programming, DFS

---

## Iteration
**Кратко:**
Iteration — это повторение действий через цикл до тех пор, пока не выполнится условие завершения.

**Полное определение:**
Итеративный подход использует конструкции `for`, `while` и аналоги, чтобы последовательно обрабатывать элементы или состояния. Во многих случаях он эквивалентен рекурсии, но обычно лучше контролирует память и поток исполнения.

**Назначение / зачем это нужно:**
Итерация нужна для явного и предсказуемого управления повторяющимися шагами алгоритма.

**Когда применять:**
Когда задача естественно формулируется как цикл, когда важна производительность и когда нужно избежать глубокой рекурсии.

**Ограничения / частые ошибки:**
Слишком сложные циклы с множеством индексов и условий могут быть менее читабельны, чем рекурсивное описание.

**Примеры:**
- Линейный проход по массиву.
- BFS через очередь.

**Связанные термины:**
Recursion, Loop, Array, Queue

---

## Memoization
**Кратко:**
Memoization — это сохранение результатов функции, чтобы не вычислять одно и то же повторно.

**Полное определение:**
При мемоизации результат для заданных аргументов кешируется, и при повторном вызове возвращается из памяти. Это особенно полезно для рекурсивных задач с перекрывающимися подзадачами.

**Назначение / зачем это нужно:**
Она нужна, чтобы снизить число повторных вычислений и ускорить алгоритмы, где одна и та же подзадача возникает многократно.

**Когда применять:**
В рекурсивных задачах, динамическом программировании сверху вниз, вычислении зависимостей, expensive pure functions.

**Ограничения / частые ошибки:**
Memoization увеличивает потребление памяти. Также ее нельзя бездумно применять к функциям с побочными эффектами или большим пространством аргументов.

**Примеры:**
- Фибоначчи с кешированием промежуточных результатов.
- Кеширование результатов разбора одних и тех же подвыражений.

**Связанные термины:**
Dynamic Programming, Cache, Recursion, Pure Function

---

## Dynamic Programming
**Кратко:**
Dynamic programming — это техника решения задач через разбиение на подзадачи и повторное использование уже найденных результатов.

**Полное определение:**
Динамическое программирование работает там, где есть оптимальная подструктура и перекрывающиеся подзадачи. Задача решается либо снизу вверх через таблицу, либо сверху вниз через recursion + memoization.

**Назначение / зачем это нужно:**
Метод нужен, чтобы заменить экспоненциальный перебор на более эффективное вычисление через хранение промежуточных состояний.

**Когда применять:**
В задачах оптимизации, подсчета путей, выравнивания строк, рюкзака, редактирования последовательностей, разбора состояний.

**Ограничения / частые ошибки:**
Частая ошибка — пытаться применить DP там, где состояние выбрано неправильно или таблица получается слишком большой. Важно уметь формализовать состояние и переход.

**Примеры:**
- Задача о рюкзаке.
- Поиск наибольшей общей подпоследовательности.

**Связанные термины:**
Memoization, Recursion, State, Optimization Problem

---

## Greedy Algorithm
**Кратко:**
Greedy algorithm — это алгоритм, который на каждом шаге принимает локально лучший выбор в надежде получить глобально хороший результат.

**Полное определение:**
Жадный подход не перебирает все варианты и не хранит сложную таблицу состояний. Он делает выбор немедленно, опираясь на текущую выгоду. Для некоторых задач это дает оптимальный результат, для других — только приближение.

**Назначение / зачем это нужно:**
Жадные алгоритмы нужны, когда требуется простое, быстрое и часто очень эффективное решение без тяжелой оптимизации.

**Когда применять:**
Когда задача обладает greedy-choice property или когда достаточно хорошего приближенного результата.

**Ограничения / частые ошибки:**
Главная ошибка — считать, что локально лучший выбор всегда приводит к глобально оптимальному. Это нужно доказывать, а не предполагать.

**Примеры:**
- Выбор интервалов по времени окончания.
- Построение минимального остовного дерева в некоторых алгоритмах.

**Связанные термины:**
Optimization, Dynamic Programming, Heuristic, Sorting
