## Краткий список терминов
- **Runtime** — среда и механизмы, обеспечивающие выполнение программы.
- **Stack** — область памяти для вызовов функций и локальных данных с предсказуемым временем жизни.
- **Heap** — область памяти для динамически создаваемых объектов.
- **Garbage Collector** — механизм автоматического освобождения неиспользуемой памяти.
- **Memory Leak** — ситуация, когда память больше не нужна программе, но продолжает оставаться занятой.
- **Pointer** — значение, содержащее адрес области памяти.
- **Reference** — ссылка на объект, обычно как более безопасная абстракция по сравнению с указателем.
- **Null Reference** — отсутствие объекта по ссылке.
- **Dangling Pointer** — указатель на память, которая уже недействительна.
- **Value Type** — тип, значение которого копируется как самостоятельная единица данных.
- **Reference Type** — тип, переменные которого обычно содержат ссылку на объект, а не сам объект.
- **Boxing** — упаковка значения value type в объектное представление.
- **Unboxing** — извлечение значения из объектного представления обратно в value type.
- **Move Semantics** — перенос владения ресурсом без лишнего копирования.
- **RAII** — управление ресурсами через время жизни объекта.
- **Allocation** — выделение памяти или другого ресурса.
- **Deallocation** — освобождение ранее выделенной памяти или ресурса.
- **Arena Allocation** — модель выделения памяти из заранее подготовленной области с массовым освобождением.

## Runtime
**Кратко:** среда и механизмы, обеспечивающие выполнение программы.

**Полное определение:**  
Runtime — это совокупность механизмов, библиотек и инфраструктуры, которые участвуют в выполнении программы после ее запуска. В зависимости от языка сюда могут входить загрузка модулей, управление памятью, обработка исключений, диспетчеризация вызовов, планирование потоков, JIT-компиляция, reflection и другие системные функции.

**Назначение:**  
Runtime снимает с прикладного кода часть низкоуровневых обязанностей и обеспечивает стандартное поведение программы в рамках конкретной платформы или языка.

**Когда применять / учитывать:**  
Runtime важно учитывать всегда, когда ты анализируешь производительность, память, запуск, исключения, асинхронность или особенности исполнения языка.

**Когда не применять / ограничения:**  
Runtime — это не отдельный паттерн и не архитектурный стиль. Это среда исполнения, а не бизнес-абстракция. Частая ошибка — путать поведение языка с поведением конкретного runtime.

**Пример:**  
В .NET runtime отвечает за GC, JIT-компиляцию, обработку исключений и выполнение IL-кода. В Go runtime управляет goroutines, scheduler и сборкой мусора.

**Связанные термины:**  
Stack, Heap, Garbage Collector, JIT, Thread, Exception Handling

## Stack
**Кратко:** область памяти для вызовов функций и локальных данных с предсказуемым временем жизни.

**Полное определение:**  
Stack — это структура памяти, в которой данные размещаются в порядке вызовов функций. Обычно в стеке хранятся параметры вызова, адрес возврата и часть локальных переменных. Освобождение происходит автоматически при выходе из функции.

**Назначение:**  
Stack обеспечивает быстрый и дешевый механизм хранения временных данных с коротким временем жизни.

**Когда применять:**  
Когда данные живут в пределах вызова функции и не должны переживать его завершение.

**Когда не применять / ограничения:**  
Stack ограничен по размеру. Слишком глубокая рекурсия или большие локальные объекты могут привести к stack overflow.

**Пример:**  
Локальная переменная-счетчик внутри функции обычно живет в стеке, если язык и компилятор не решили иначе.

**Связанные термины:**  
Heap, Allocation, Recursion, Stack Overflow

## Heap
**Кратко:** область памяти для динамически создаваемых объектов.

**Полное определение:**  
Heap — это область памяти, из которой выделяются объекты с динамическим временем жизни. Такие объекты могут существовать дольше одного вызова функции и часто доступны из разных частей программы по ссылке или указателю.

**Назначение:**  
Heap позволяет хранить объекты, размер или время жизни которых нельзя удобно определить на этапе компиляции или в границах одного вызова.

**Когда применять:**  
Когда объект должен жить дольше текущего scope, передаваться по ссылке, иметь динамический размер или участвовать в сложных графах объектов.

**Когда не применять / ограничения:**  
Heap-аллоцирование обычно дороже, чем работа со стеком. Избыточные аллокации увеличивают нагрузку на allocator и GC.

**Пример:**  
Экземпляр класса в C# или объект, созданный через `new` в большинстве языков с managed memory, обычно размещается в heap.

**Связанные термины:**  
Stack, Garbage Collector, Allocation, Reference Type

## Garbage Collector
**Кратко:** механизм автоматического освобождения неиспользуемой памяти.

**Полное определение:**  
Garbage Collector, или GC, — это подсистема runtime, которая определяет объекты, больше недостижимые из живых ссылок, и освобождает занятую ими память. Конкретные алгоритмы отличаются: mark-and-sweep, generational GC, concurrent GC и другие.

**Назначение:**  
GC уменьшает риск ручных ошибок управления памятью и упрощает разработку за счет автоматического контроля жизненного цикла памяти.

**Когда применять / учитывать:**  
В языках с managed memory нужно учитывать влияние GC на latency, throughput и общую форму аллокаций.

**Когда не применять / ограничения:**  
GC не решает все проблемы с памятью. Он не спасает от удержания ненужных ссылок, из-за чего объект остается достижимым и не собирается. Также GC может вносить паузы или дополнительную нагрузку.

**Пример:**  
Если коллекция перестала ссылаться на объект, а других ссылок нет, GC сможет освободить память, занятую этим объектом.

**Связанные термины:**  
Heap, Memory Leak, Reachability, Runtime

## Memory Leak
**Кратко:** ситуация, когда память больше не нужна программе, но продолжает оставаться занятой.

**Полное определение:**  
Memory leak — это потеря контроля над памятью или ресурсами, при которой они не освобождаются вовремя. В unmanaged-средах это часто означает забытое освобождение памяти. В managed-средах leak чаще связан не с отсутствием `free`, а с тем, что объект все еще достижим по ссылкам, хотя бизнес-логике он уже не нужен.

**Назначение термина:**  
Термин помогает отделить нормальное потребление памяти от ошибочного удержания данных.

**Когда учитывать:**  
При анализе долгоживущих процессов, сервисов, фоновых задач, кешей, событий, подписок и large object allocations.

**Когда не путать:**  
Не каждое увеличение памяти — утечка. Возможно, это нормальный кеш, отложенное освобождение GC или пиковая нагрузка.

**Пример:**  
Сервис подписался на события singleton-объекта, но не отписался. Из-за этого экземпляры остаются достижимыми и не собираются GC.

**Связанные термины:**  
Garbage Collector, Reference, Event Subscription, Resource Leak

## Pointer
**Кратко:** значение, содержащее адрес области памяти.

**Полное определение:**  
Pointer — это низкоуровневая конструкция, которая хранит адрес данных в памяти и позволяет обращаться к ним напрямую. Указатели дают большой контроль, но вместе с ним — высокую цену ошибки.

**Назначение:**  
Указатели применяются для прямой работы с памятью, эффективной передачи доступа к данным, реализации низкоуровневых структур, системного программирования и interop.

**Когда применять:**  
Когда нужен прямой контроль над памятью, взаимодействие с C ABI, работа с буферами, unsafe-оптимизации или системное программирование.

**Когда не применять / ограничения:**  
Без необходимости указатели лучше не использовать. Они повышают риск dangling pointers, buffer overrun, memory corruption и трудноуловимых багов.

**Пример:**  
В C переменная `int* p` может указывать на адрес целого числа и позволять читать или изменять его по адресу.

**Связанные термины:**  
Reference, Dangling Pointer, Allocation, Unsafe Code

## Reference
**Кратко:** ссылка на объект, обычно как более безопасная абстракция по сравнению с указателем.

**Полное определение:**  
Reference — это механизм доступа к объекту не напрямую по значению, а через косвенную связь с ним. В одних языках reference является явной языковой сущностью, в других — моделью, скрывающей детали указателей и управления памятью.

**Назначение:**  
Reference позволяет нескольким переменным обращаться к одному объекту без полного копирования его состояния.

**Когда применять:**  
Когда объект должен совместно использоваться, передаваться без копии или иметь идентичность, отличную от простого значения.

**Когда не применять / ограничения:**  
Reference semantics усложняют reasoning о мутациях: изменение объекта через одну ссылку видно через другие. Частая ошибка — ожидать copy semantics там, где работают ссылки.

**Пример:**  
Если две переменные ссылаются на один и тот же объект списка, изменение через одну переменную будет видно через другую.

**Связанные термины:**  
Pointer, Reference Type, Alias, Immutability

## Null Reference
**Кратко:** отсутствие объекта по ссылке.

**Полное определение:**  
Null reference обозначает состояние, при котором переменная ссылочного типа не указывает ни на один объект. Это специальное значение, сигнализирующее об отсутствии привязанного экземпляра.

**Назначение:**  
Null используется как маркер отсутствия значения там, где тип допускает такую модель.

**Когда применять:**  
Осторожно, только когда отсутствие значения — нормальная часть модели и это явно отражено в API.

**Когда не применять / ограничения:**  
Чрезмерное использование null делает API хрупким. Во многих случаях лучше использовать Option/Maybe-подобные модели, nullable annotations или явные result-объекты.

**Пример:**  
Метод поиска может вернуть `null`, если запись не найдена, но более надежным вариантом часто будет `TryGet`-паттерн или result type.

**Связанные термины:**  
Reference, Nullable, Null Safety, Optional

## Dangling Pointer
**Кратко:** указатель на память, которая уже недействительна.

**Полное определение:**  
Dangling pointer — это указатель, который продолжает хранить старый адрес после того, как соответствующая память уже освобождена или объект завершил свое существование. Обращение по такому адресу приводит к неопределенному поведению.

**Назначение термина:**  
Термин фиксирует один из самых опасных классов ошибок ручного управления памятью.

**Когда учитывать:**  
В C, C++, Rust через unsafe и в любых low-level сценариях, где жизненный цикл памяти контролируется вручную или почти вручную.

**Когда не путать:**  
Dangling pointer — это не null pointer. Null указывает на отсутствие объекта, dangling pointer указывает на уже невалидную память.

**Пример:**  
Функция вернула указатель на локальную переменную. После выхода из функции этот адрес больше невалиден.

**Связанные термины:**  
Pointer, Use-After-Free, Undefined Behavior, Lifetime

## Value Type
**Кратко:** тип, значение которого копируется как самостоятельная единица данных.

**Полное определение:**  
Value type — это тип, при присваивании или передаче которого обычно копируется само значение, а не ссылка на общий объект. Такие типы хорошо подходят для компактных и семантически самостоятельных данных.

**Назначение:**  
Value semantics упрощает reasoning о данных, потому что изменения одной копии не влияют на другую.

**Когда применять:**  
Для небольших, логически атомарных структур: координат, дат, денежных значений, идентификаторов, immutable records.

**Когда не применять / ограничения:**  
Крупные value types могут быть дорогими при копировании. Также не стоит насильно делать ссылочные сущности value type только ради "чистоты" модели.

**Пример:**  
Структура `Point(x, y)` как value type удобна: ее можно копировать без риска неожиданной общей мутации.

**Связанные термины:**  
Reference Type, Copy Semantics, Immutability

---

## Reference Type
**Кратко:** тип, переменные которого обычно содержат ссылку на объект, а не сам объект.

**Полное определение:**  
Reference type — это тип, значение переменной которого обычно представляет доступ к объекту, расположенному отдельно в памяти. Несколько переменных могут указывать на один и тот же объект.

**Назначение:**  
Reference types подходят для сущностей с идентичностью, разделяемым состоянием, сложным жизненным циклом и графами зависимостей.

**Когда применять:**  
Для доменных сущностей, сервисов, изменяемых коллекций, объектов инфраструктуры и состояний, которые должны разделяться.

**Когда не применять / ограничения:**  
Reference semantics повышают риск неявных мутаций, aliasing bugs и сложности при конкурентном доступе.

**Пример:**  
Экземпляр класса `Order` может передаваться через разные слои приложения как один и тот же объект со своей идентичностью.

**Связанные термины:**  
Value Type, Reference, Identity, Heap

## Boxing and Unboxing
**Кратко:** преобразование значения в объектное представление и обратно.

**Полное определение:**  
Boxing — это помещение значения value type в объектную оболочку, чтобы его можно было использовать там, где ожидается ссылочный или общий объектный тип. Unboxing — обратное извлечение исходного значения. Часто такие операции влекут аллокации и дополнительные проверки типов.

**Назначение:**  
Позволяет унифицировать работу с разными типами через общий интерфейс или базовый объектный тип.

**Когда применять:**  
Обычно не как цель, а как побочный механизм платформы. Разработчик должен скорее замечать и минимизировать лишний boxing в hot paths.

**Когда не применять / ограничения:**  
Избыточный boxing ухудшает производительность и порождает лишние аллокации.

**Пример:**  
Передача числового значения в API, ожидающий `object`, может вызвать boxing.

**Связанные термины:**  
Value Type, Reference Type, Allocation, Performance

## Move Semantics
**Кратко:** перенос владения ресурсом без лишнего копирования.

**Полное определение:**  
Move semantics — это модель, при которой ресурс не копируется, а передается от одного владельца к другому. После переноса исходный объект остается в допустимом, но уже "перемещенном" состоянии. Это позволяет избежать дорогостоящего копирования больших структур или ресурсов.

**Назначение:**  
Снижает накладные расходы на создание и передачу объектов, особенно владеющих памятью, файлами, буферами или сокетами.

**Когда применять:**  
В системах с явным управлением ресурсами и в языках, где move semantics является частью модели владения.

**Когда не применять / ограничения:**  
Нужно четко понимать состояние объекта после move. Ошибки здесь часто связаны с повторным использованием уже перемещенного значения.

**Пример:**  
В C++ временный буфер можно переместить в контейнер без полного копирования содержимого.

**Связанные термины:**  
Ownership, Copy Semantics, RAII, Resource Management

---

## RAII
**Кратко:** управление ресурсами через время жизни объекта.

**Полное определение:**  
RAII, Resource Acquisition Is Initialization, — идиома, при которой захват ресурса выполняется в конструкторе объекта, а освобождение — в деструкторе или эквивалентном механизме завершения времени жизни. Ресурс оказывается привязан к области видимости и жизненному циклу объекта.

**Назначение:**  
RAII делает освобождение ресурсов надежным и предсказуемым даже при исключениях и ранних выходах из функции.

**Когда применять:**  
Для файлов, mutex, сокетов, памяти, транзакций и других ресурсов, требующих гарантированного освобождения.

**Когда не применять / ограничения:**  
В языках без детерминированного разрушения объектов RAII в исходном виде недоступен, и его приходится моделировать через `using`, `defer`, `try/finally` и аналоги.

**Пример:**  
Объект-обертка для файла открывает дескриптор в конструкторе и закрывает его автоматически при выходе из scope.

**Связанные термины:**  
Lifetime, Resource Management, Move Semantics, Deterministic Cleanup

---

## Allocation
**Кратко:** выделение памяти или другого ресурса.

**Полное определение:**  
Allocation — это операция резервирования памяти или иного ограниченного ресурса для дальнейшего использования программой. Чаще всего под allocation понимают именно выделение памяти, но концептуально это шире.

**Назначение:**  
Выделение позволяет создать область, где объект или данные смогут существовать и использоваться в рамках программы.

**Когда применять:**  
Когда нужен новый объект, буфер, коллекция, соединение или другой ресурс с контролируемым жизненным циклом.

**Когда не применять / ограничения:**  
Чрезмерные аллокации часто становятся источником деградации производительности, GC pressure и фрагментации памяти.

**Пример:**  
Создание новой строки, списка или объекта может вызвать allocation в heap.

**Связанные термины:**  
Deallocation, Heap, Stack, Arena Allocation

## Deallocation
**Кратко:** освобождение ранее выделенной памяти или ресурса.

**Полное определение:**  
Deallocation — это возврат памяти или другого ресурса системе, allocator-у или runtime после того, как он больше не нужен. В одних языках это делается вручную, в других — автоматически или полуавтоматически.

**Назначение:**  
Освобождение предотвращает утечки и делает ресурс доступным для повторного использования.

**Когда применять:**  
Всегда, когда модель языка или платформы требует явного освобождения памяти либо других ресурсов вроде файлов, соединений, блокировок.

**Когда не применять / ограничения:**  
Нельзя освобождать ресурс дважды и нельзя использовать его после освобождения. Это частый источник критических ошибок.

**Пример:**  
В unmanaged-коде память, полученная через allocator, должна быть корректно возвращена обратно.

**Связанные термины:**  
Allocation, Memory Leak, Dangling Pointer, RAII

## Arena Allocation
**Кратко:** модель выделения памяти из заранее подготовленной области с массовым освобождением.

**Полное определение:**  
Arena allocation — это техника, при которой множество объектов выделяется из общего буфера или региона памяти, а освобождение производится сразу для всей арены, а не по одному объекту. Это сильно упрощает и ускоряет работу с краткоживущими наборами объектов.

**Назначение:**  
Снижает накладные расходы на индивидуальные allocation/deallocation и помогает контролировать локальность данных.

**Когда применять:**  
Для парсеров, компиляторов, временных деревьев, request-scoped данных и других сценариев, где много объектов живут примерно одинаковое время.

**Когда не применять / ограничения:**  
Плохо подходит, если объектам нужен сильно разный жизненный цикл. Индивидуально освободить отдельный объект из арены обычно нельзя или это невыгодно.

**Пример:**  
Парсер выделяет все узлы AST текущего файла в одной арене, а после завершения разбора освобождает арену целиком.

**Связанные термины:**  
Allocation, Heap, Lifetime, Performance
