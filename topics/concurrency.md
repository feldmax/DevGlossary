## Краткий список терминов
- **Concurrency** — независимое продвижение нескольких задач с возможным чередованием выполнения.
- **Parallelism** — реальное одновременное выполнение нескольких задач на разных вычислительных ресурсах.
- **Thread** — поток выполнения внутри процесса.
- **Process** — отдельный экземпляр программы со своим адресным пространством и ресурсами.
- **Coroutine** — легковесная единица выполнения, которая может приостанавливать и возобновлять работу.
- **Async/Await** — модель неблокирующего ожидания асинхронных операций.
- **Event Loop** — цикл, который принимает события и поочередно исполняет связанные задачи.
- **Channel** — способ безопасно передавать данные между конкурентными исполнителями.
- **Shared State** — общее состояние, к которому обращаются несколько исполнителей.
- **Mutex** — примитив взаимного исключения для защиты критической секции.
- **Semaphore** — примитив синхронизации, ограничивающий число одновременных доступов.
- **Atomic Operation** — операция, которая выполняется как единое неделимое действие.
- **Race Condition** — ошибка, при которой результат зависит от непредсказуемого порядка выполнения.
- **Deadlock** — взаимная блокировка, при которой участники ждут друг друга бесконечно.
- **Livelock** — состояние, при котором участники активны, но прогресса нет.
- **Starvation** — ситуация, когда задача слишком долго не получает доступ к ресурсу.
- **Thread Safety** — свойство кода корректно работать при многопоточном доступе.
- **Backpressure** — механизм ограничения потока данных при перегрузке потребителя.

## Concurrency
**Кратко:**
Concurrency — это организация системы так, чтобы несколько задач могли продвигаться независимо друг от друга, даже если в каждый конкретный момент реально исполняется только одна из них.

**Полное определение:**
Конкурентность описывает способ структурировать выполнение работы, когда в системе одновременно есть несколько активных задач. Эти задачи могут чередоваться, приостанавливаться, ожидать внешних событий и продолжаться позже. Главное здесь не физическая одновременность, а управление независимым прогрессом нескольких операций.

**Назначение / зачем это нужно:**
Concurrency нужна, чтобы эффективно обслуживать ввод-вывод, UI, сетевые запросы, очереди, фоновые работы и другие сценарии, где простаивание CPU или блокировка одного пути не должны останавливать остальную систему.

**Когда применять:**
Когда в приложении есть несколько независимых операций: обработка множества запросов, фоновые джобы, асинхронный I/O, взаимодействие с несколькими внешними сервисами.

**Ограничения / частые ошибки:**
Частая ошибка — путать concurrency с parallelism. Конкурентная система может быть однопоточной. Еще одна ошибка — думать, что concurrency автоматически ускоряет CPU-bound код.

**Примеры:**
- Web-сервер принимает новые соединения, пока старые запросы ждут ответа от базы данных.
- UI-приложение продолжает реагировать на действия пользователя, пока выполняется загрузка данных.

**Связанные термины:**
Parallelism, Async/Await, Event Loop, Thread, Process

## Parallelism
**Кратко:**
Parallelism — это реальное одновременное выполнение нескольких задач на разных ядрах CPU или других вычислительных ресурсах.

**Полное определение:**
Параллелизм означает, что несколько операций выполняются физически одновременно. В отличие от concurrency, где задачи могут просто чередоваться, parallelism использует несколько вычислительных единиц, чтобы реально выполнять работу в один момент времени.

**Назначение / зачем это нужно:**
Параллелизм нужен для ускорения вычислений, особенно в CPU-bound задачах: обработке данных, численных расчетах, рендеринге, аналитике, компиляции.

**Когда применять:**
Когда задача разбивается на независимые части и узкое место — именно вычисления, а не ввод-вывод.

**Ограничения / частые ошибки:**
Не всякую задачу выгодно распараллеливать. Есть накладные расходы на координацию, синхронизацию, обмен данными и распределение работы. Маленькие задачи могут работать медленнее из-за этих издержек.

**Примеры:**
- Обработка большого массива данных по частям на нескольких ядрах.
- Параллельная компиляция нескольких модулей проекта.

**Связанные термины:**
Concurrency, Thread, Process, CPU-bound, Synchronization

## Thread
**Кратко:**
Thread — поток выполнения внутри процесса, разделяющий с другими потоками память и ресурсы процесса.

**Полное определение:**
Поток — это отдельная последовательность инструкций внутри одного процесса. Несколько потоков одного процесса используют общее адресное пространство, поэтому могут быстро обмениваться данными, но при этом создают риск гонок и конфликтов доступа.

**Назначение / зачем это нужно:**
Потоки позволяют одновременно выполнять несколько частей одной программы, использовать многоядерность и отделять независимые направления работы.

**Когда применять:**
Когда нужно параллельное выполнение внутри одного приложения или конкурентная обработка нескольких задач с доступом к общей памяти.

**Ограничения / частые ошибки:**
Потоки дороги по сравнению с coroutines или async tasks. Частая ошибка — создавать слишком много потоков или делить между ними изменяемое состояние без синхронизации.

**Примеры:**
- Один поток обслуживает UI, другой выполняет фоновые вычисления.
- Пул потоков обрабатывает входящие задания.

**Связанные термины:**
Process, Mutex, Thread Safety, Context Switch, Parallelism

## Process
**Кратко:**
Process — это отдельный экземпляр программы со своим адресным пространством, дескрипторами и системными ресурсами.

**Полное определение:**
Процесс изолирован от других процессов на уровне памяти и многих системных ресурсов. Внутри процесса может быть один или несколько потоков. Межпроцессное взаимодействие обычно сложнее и дороже, чем обмен данными между потоками, но дает более сильную изоляцию.

**Назначение / зачем это нужно:**
Процессы позволяют изолировать сбои, разграничивать ресурсы, запускать независимые программы и строить более безопасные архитектуры.

**Когда применять:**
Когда нужна изоляция памяти, отдельные права доступа, независимое завершение или четкое разделение компонентов.

**Ограничения / частые ошибки:**
Общение между процессами сложнее: нужны сокеты, пайпы, shared memory, RPC или другие IPC-механизмы. Ошибка — использовать процессы там, где достаточно одного процесса с пулом потоков.

**Примеры:**
- Браузер запускает вкладки или сервисы в отдельных процессах.
- Несколько контейнеров обмениваются данными по сети как отдельные процессы.

**Связанные термины:**
Thread, Isolation, IPC, Parallelism, Process Boundary

## Coroutine
**Кратко:**
Coroutine — легковесная единица выполнения, которая умеет приостанавливать и возобновлять работу без тяжелого системного потока.

**Полное определение:**
Короутина — это форма кооперативного выполнения, где задача может временно уступить управление и продолжить выполнение позже. Конкретная реализация зависит от языка и runtime: goroutines в Go, coroutines в Kotlin, async tasks в C# и Python имеют разные модели, но идея похожа — легковесное управление множеством операций.

**Назначение / зачем это нужно:**
Coroutines позволяют обслуживать большое количество конкурентных задач с меньшими накладными расходами, чем при создании отдельного системного потока на каждую задачу.

**Когда применять:**
Когда много сетевых или I/O-bound операций, когда важно держать высокую конкурентность без большого числа OS threads.

**Ограничения / частые ошибки:**
Короутины не отменяют необходимость синхронизации, если они используют shared state. Ошибка — считать их «бесплатными» и бесконтрольно плодить без учета памяти, очередей и backpressure.

**Примеры:**
- Тысячи сетевых запросов обслуживаются через легковесные задачи runtime.
- Асинхронная функция ждет ответ БД, освобождая исполнителя для других задач.

**Связанные термины:**
Async/Await, Event Loop, Thread, Scheduler, Concurrency

## Async/Await
**Кратко:**
Async/Await — способ писать асинхронный код в последовательном стиле без явного ручного управления callback-цепочками.

**Полное определение:**
Модель async/await позволяет функции временно приостанавливать выполнение на ожидании асинхронной операции и продолжать его позже, когда результат готов. При этом поток часто не блокируется: управление возвращается runtime или event loop.

**Назначение / зачем это нужно:**
Подход упрощает асинхронный код, делает его более читаемым и помогает строить высококонкурентные I/O-системы без блокировки потоков.

**Когда применять:**
Когда операция зависит от сети, файловой системы, базы данных, таймеров или других медленных внешних источников.

**Ограничения / частые ошибки:**
Частая ошибка — вызывать async-код и затем блокироваться на результате синхронно. Еще одна ошибка — ожидать ускорения CPU-bound кода только из-за `async`.

**Примеры:**
- HTTP-запрос ожидается через `await`, а поток обрабатывает другие задачи.
- Асинхронная запись в базу выполняется без блокировки UI.

**Связанные термины:**
Coroutine, Event Loop, Future, Promise, Concurrency

## Event Loop
**Кратко:**
Event Loop — цикл, который получает события, планирует задачи и вызывает нужные обработчики по мере готовности работы.

**Полное определение:**
Event loop — это центральный механизм некоторых runtime, особенно в асинхронных системах. Он следит за готовностью операций ввода-вывода, таймерами, очередями задач и по очереди запускает соответствующие continuation или callbacks.

**Назначение / зачем это нужно:**
Позволяет одному или нескольким исполнителям эффективно обслуживать множество I/O-bound задач без отдельного потока на каждую операцию.

**Когда применять:**
В event-driven приложениях, сетевых серверах, UI-приложениях, JavaScript-runtime, Python async-средах и других системах с неблокирующим I/O.

**Ограничения / частые ошибки:**
Блокирующая операция внутри event loop может заморозить весь цикл. Частая ошибка — выполнять тяжелые CPU-bound вычисления прямо в loop.

**Примеры:**
- JavaScript runtime в браузере или Node.js обрабатывает очередь событий через event loop.
- Async web-сервер переключается между тысячами подключений по готовности сокетов.

**Связанные термины:**
Async/Await, Coroutine, Non-blocking I/O, Callback, Scheduler

## Channel
**Кратко:**
Channel — это средство передачи данных между конкурентными исполнителями через сообщения, а не через общий изменяемый объект.

**Полное определение:**
Канал представляет собой очередь или синхронизированную точку обмена, через которую одна задача отправляет данные, а другая их получает. Идея особенно популярна в CSP-подобных моделях: «не делить память между потоками, а передавать владение данными через сообщения».

**Назначение / зачем это нужно:**
Channels помогают уменьшить объем shared state и упростить reasoning о порядке взаимодействия между задачами.

**Когда применять:**
Когда одна часть системы производит данные, а другая потребляет: pipeline, worker pool, fan-in/fan-out, обработка событий.

**Ограничения / частые ошибки:**
Канал не решает автоматически все проблемы координации. Можно получить deadlock, переполнение буфера, зависание consumer или producer. Важно продумывать завершение, закрытие канала и backpressure.

**Примеры:**
- Один worker читает задания из канала, другие отправляют в него работы.
- Pipeline из нескольких стадий передает результаты по каналам.

**Связанные термины:**
Message Passing, Backpressure, Worker Pool, Shared State, Queue

## Shared State
**Кратко:**
Shared state — это данные, к которым одновременно имеют доступ несколько потоков, задач или компонентов.

**Полное определение:**
Общее состояние возникает, когда несколько исполнителей читают и/или изменяют один и тот же объект, структуру или набор данных. Это мощный, но опасный механизм, потому что порядок доступа начинает влиять на корректность программы.

**Назначение / зачем это нужно:**
Иногда shared state неизбежен: общий кеш, счетчики, карта сессий, внутреннее состояние объекта, разделяемый буфер.

**Когда применять:**
Когда совместный доступ действительно нужен и цена копирования, изоляции или message passing слишком высока.

**Ограничения / частые ошибки:**
Неуправляемый shared state — главный источник race conditions. Частая ошибка — добавлять общий mutable объект просто из удобства, а потом пытаться «допридумать» синхронизацию постфактум.

**Примеры:**
- Несколько потоков обновляют общий счетчик.
- Worker pool пишет результаты в одну общую map.

**Связанные термины:**
Mutex, Atomic Operation, Race Condition, Immutability, Channel

## Mutex
**Кратко:**
Mutex — примитив взаимного исключения, который гарантирует, что критическая секция выполняется только одним участником одновременно.

**Полное определение:**
Mutex позволяет сериализовать доступ к разделяемому ресурсу. Пока один поток или задача удерживает mutex, остальные должны ждать освобождения. Это один из базовых инструментов синхронизации в многопоточном коде.

**Назначение / зачем это нужно:**
Mutex защищает инварианты общего состояния и предотвращает гонки при чтении и записи.

**Когда применять:**
Когда несколько исполнителей могут одновременно менять или читать изменяемые данные и нужно гарантировать атомарность более крупной последовательности операций.

**Ограничения / частые ошибки:**
Ошибки включают слишком широкие критические секции, вложенные lock'и, забытый unlock, lock order inversion и чрезмерную конкуренцию за один mutex.

**Примеры:**
- Защита словаря, в который несколько потоков записывают результаты.
- Обновление нескольких связанных полей объекта под одним lock.

**Связанные термины:**
Semaphore, Atomic Operation, Deadlock, Thread Safety, Shared State

## Semaphore
**Кратко:**
Semaphore — примитив синхронизации, который позволяет одновременно пропускать ограниченное число участников.

**Полное определение:**
Семафор хранит счетчик доступных разрешений. Когда задача входит, счетчик уменьшается; когда выходит — увеличивается. Если разрешений нет, новые участники ждут. В отличие от mutex, семафор допускает не одного, а N одновременных участников.

**Назначение / зачем это нужно:**
Нужен для ограничения параллелизма, защиты дефицитных ресурсов и управления нагрузкой.

**Когда применять:**
Когда нужно, например, не более 10 одновременных запросов к внешнему API или ограничение числа активных задач в пуле.

**Ограничения / частые ошибки:**
Семафор легко использовать неверно: забыть release, взять больше разрешений, чем нужно, или использовать его там, где нужен именно mutex для защиты инварианта.

**Примеры:**
- Ограничение числа одновременных подключений к ресурсу.
- Ограничение количества параллельных фоновых операций.

**Связанные термины:**
Mutex, Rate Limiting, Backpressure, Resource Pool, Concurrency Limit

## Atomic Operation
**Кратко:**
Atomic operation — операция, которую другие участники не могут наблюдать в промежуточном состоянии.

**Полное определение:**
Атомарная операция выполняется как единый неделимый шаг с точки зрения других потоков или исполнителей. Это может быть чтение, запись, compare-and-swap, increment и другие низкоуровневые примитивы, поддерживаемые CPU и runtime.

**Назначение / зачем это нужно:**
Позволяет реализовывать простые lock-free механизмы, счетчики, флаги и базовые протоколы синхронизации без тяжелых блокировок.

**Когда применять:**
Когда нужно атомарно изменить простое значение или построить более сложный примитив синхронизации.

**Ограничения / частые ошибки:**
Атомарность одной операции не делает автоматически атомарной целую последовательность из нескольких действий. Частая ошибка — защищать сложный объект только отдельными atomic-полями и терять инварианты.

**Примеры:**
- Атомарный инкремент счетчика запросов.
- Compare-and-swap для смены состояния без mutex.

**Связанные термины:**
Mutex, Lock-free, CAS, Memory Ordering, Race Condition

## Race Condition
**Кратко:**
Race condition — это ошибка, при которой результат зависит от того, в каком порядке и в какой момент выполняются конкурентные операции.

**Полное определение:**
Гонка возникает, когда несколько исполнителей обращаются к общим данным без корректной координации, и допустимый порядок их выполнения не определен. Из-за этого программа может иногда работать правильно, а иногда — нет, причем воспроизвести проблему бывает трудно.

**Назначение термина:**
Термин помогает точно описывать класс конкурентных багов, связанных не с бизнес-логикой, а с небезопасной координацией исполнения.

**Когда учитывать:**
Всегда, когда есть shared mutable state, многопоточность, async callbacks, event handlers, фоновые задачи и неблокирующий код.

**Когда не путать:**
Не каждая nondeterministic система имеет race condition. Если порядок не фиксирован, но корректность от него не зависит, гонки может и не быть.

**Примеры:**
- Два потока одновременно увеличивают счетчик без atomic или mutex.
- Один поток читает объект, пока другой меняет его поля без синхронизации.

**Связанные термины:**
Shared State, Mutex, Atomic Operation, Thread Safety, Data Race

## Deadlock
**Кратко:**
Deadlock — ситуация, при которой несколько участников навсегда блокируют друг друга, ожидая освобождения ресурсов.

**Полное определение:**
В deadlock каждый участник держит часть ресурсов и ждет другую часть, которую удерживает кто-то еще. Так как никто не может продвинуться, система застывает. Это классическая проблема многопоточного и распределенного кода.

**Назначение термина:**
Позволяет точно описывать взаимную блокировку, отличая ее от обычной медленной работы или высоких задержек.

**Когда учитывать:**
Когда используются mutex, семафоры, блокирующие очереди, sync-over-async паттерны, nested locks и другие механизмы ожидания.

**Ограничения / частые ошибки:**
Часто deadlock появляется из-за разного порядка захвата lock'ов, забытых release или блокировки на результате async-операции в неподходящем контексте.

**Примеры:**
- Поток A держит lock X и ждет Y, поток B держит Y и ждет X.
- UI-поток блокируется на async-результате, а continuation этого async должна выполниться на том же UI-потоке.

**Связанные термины:**
Mutex, Semaphore, Livelock, Blocking, Synchronization

## Livelock
**Кратко:**
Livelock — состояние, при котором участники не заблокированы, но постоянно реагируют друг на друга и не совершают полезного прогресса.

**Полное определение:**
В livelock система остается активной: потоки или процессы выполняются, повторяют проверки, откатывают действия, уступают друг другу, но итоговая полезная работа не движется вперед. Внешне это выглядит как жизнь без результата.

**Назначение термина:**
Позволяет отделить взаимную блокировку от неэффективной «суеты», когда приложение занято, но по сути стоит на месте.

**Когда учитывать:**
В алгоритмах повторных попыток, конкурентных протоколах, системах с optimistic locking и сложной координацией реакций между участниками.

**Ограничения / частые ошибки:**
Часто livelock возникает, когда все участники используют одинаковую стратегию retry без рандомизации или backoff.

**Примеры:**
- Два потока видят конфликт, оба одновременно отступают и снова одновременно повторяют попытку.
- Два робота уступают дорогу друг другу синхронно и бесконечно меняют направление.

**Связанные термины:**
Deadlock, Retry, Backoff, Starvation, Optimistic Concurrency

## Starvation
**Кратко:**
Starvation — это ситуация, когда задача слишком долго не получает ресурс, процессорное время или доступ к выполнению.

**Полное определение:**
При starvation система в целом может работать, но отдельная задача постоянно проигрывает конкуренцию за ресурсы: lock, scheduler time slice, очередь, I/O capacity или другой ограниченный ресурс. В результате задача формально жива, но практически не продвигается.

**Назначение термина:**
Термин нужен, чтобы различать общую перегрузку системы и несправедливое распределение ресурсов между конкретными участниками.

**Когда учитывать:**
В scheduler'ах, приоритетных очередях, lock contention, unfair mutex-реализациях, aggressive retry loops и worker pool'ах.

**Ограничения / частые ошибки:**
Ошибка — считать, что раз система «в среднем» работает, то проблем нет. Starvation может долго не замечаться, но ломать tail latency и SLA отдельных операций.

**Примеры:**
- Низкоприоритетная задача почти никогда не получает CPU.
- Один worker постоянно забирает задания, а другой простаивает.

**Связанные термины:**
Scheduler, Fairness, Deadlock, Livelock, Backpressure

## Thread Safety
**Кратко:**
Thread safety — это свойство кода сохранять корректность при одновременном доступе из нескольких потоков.

**Полное определение:**
Код считается thread-safe, если он остается корректным при конкурентном выполнении: не ломает инварианты, не теряет данные и не попадает в гонки из-за межпоточного взаимодействия. Это может достигаться через immutability, synchronization, message passing, atomics или комбинацию подходов.

**Назначение / зачем это нужно:**
Thread safety нужна для надежной работы библиотек, сервисов, кешей, инфраструктурных компонентов и любых общих объектов в многопоточной среде.

**Когда применять:**
Когда компонент используется из нескольких потоков или потенциально может использоваться параллельно.

**Ограничения / частые ошибки:**
Thread-safe код не всегда быстрый и не всегда простой. Ошибка — делать API формально потокобезопасным, но неудобным или слишком дорогим по производительности.

**Примеры:**
- Concurrent collection, корректно работающая при чтении и записи из нескольких потоков.
- Immutable объект, который можно безопасно разделять между потоками без lock.

**Связанные термины:**
Race Condition, Mutex, Atomic Operation, Immutability, Shared State

## Backpressure
**Кратко:**
Backpressure — механизм, который не позволяет производителю бесконтрольно засыпать потребителя данными быстрее, чем тот успевает их обрабатывать.

**Полное определение:**
Backpressure — это форма управления потоком данных в конкурентных и распределенных системах. Если producer быстрее consumer, очередь начинает расти, увеличивается память, latency и риск деградации. Backpressure вводит ограничение: замедляет producer, ограничивает буферы, сбрасывает часть нагрузки или перераспределяет работу.

**Назначение / зачем это нужно:**
Позволяет системе деградировать контролируемо, а не через бесконечный рост очередей, памяти и времени отклика.

**Когда применять:**
В pipeline, message queues, streaming systems, async processing, worker pools, API clients и любых producer-consumer сценариях.

**Ограничения / частые ошибки:**
Ошибка — полагаться только на большие очереди и считать это решением. Без backpressure очередь просто скрывает проблему до момента перегрузки.

**Примеры:**
- Ограничение числа заданий в очереди с ожиданием свободного места.
- Снижение скорости чтения из сокета, пока downstream не успевает обработать данные.

**Связанные термины:**
Channel, Semaphore, Rate Limiting, Throughput, Graceful Degradation
