## Краткий список терминов
- **OOP** — подход к проектированию, в котором программа строится вокруг объектов с состоянием и поведением.
- **Class** — описание структуры и поведения объектов одного типа.
- **Object** — конкретный экземпляр класса со своим состоянием.
- **Abstraction** — выделение существенных свойств модели и сокрытие лишних деталей реализации.
- **Encapsulation** — объединение данных и логики работы с ними с контролируемым доступом извне.
- **Inheritance** — механизм, при котором один класс расширяет или специализирует другой.
- **Composition** — способ строить сложное поведение из вложенных объектов и зависимостей.
- **Aggregation** — более слабая форма отношения «содержит», где части могут жить отдельно от целого.
- **Polymorphism** — возможность работать с разными реализациями через общий интерфейс или базовый тип.
- **Interface** — контракт, описывающий доступное поведение без реализации.
- **Abstract Class** — базовый класс с общей логикой и неполной реализацией.
- **Contract** — набор ожидаемых правил поведения компонента или типа.
- **Subtype** — тип, который допустимо использовать вместо другого типа в том же контексте.
- **Liskov Substitution Principle** — подтип должен быть взаимозаменяем с базовым типом без поломки поведения.
- **Method Overriding** — переопределение метода базового класса в наследнике.
- **Method Overloading** — несколько методов с одним именем, но разными сигнатурами.
- **Generics** — параметризация типов для повторного использования кода без потери типобезопасности.

## OOP
**Кратко:**
OOP — это способ организовать код вокруг объектов, которые объединяют состояние и поведение и взаимодействуют друг с другом через понятные контракты.

**Полное определение:**
Object-Oriented Programming — это парадигма, в которой основные строительные элементы программы — объекты. Каждый объект обычно имеет состояние, хранимое в полях, и поведение, выраженное методами. На практике OOP используют не ради самих классов, а ради моделирования домена, ограничения ответственности, переиспользования кода и контроля сложности.

**Назначение / зачем это нужно:**
OOP помогает описывать систему через сущности предметной области, группировать связанные данные и поведение и строить расширяемые модели через интерфейсы и композицию.

**Когда применять:**
Подходит для бизнес-приложений, UI, доменных моделей, сервисов и библиотек, где важно явно описывать роли, состояния и правила взаимодействия между частями системы.

**Ограничения / частые ошибки:**
Частая ошибка — считать, что OOP автоматически означает хорошую архитектуру. Плохо спроектированная объектная модель может быть перегруженной, чрезмерно иерархической и трудной в сопровождении. Еще одна ошибка — использовать наследование там, где композиция проще и безопаснее.

**Примеры:**
- `Order` хранит состояние заказа и методы, связанные с его жизненным циклом.
- `INotificationSender` позволяет работать с email, SMS и push как с единым контрактом.

**Связанные термины:**
Class, Object, Encapsulation, Composition, Interface, Polymorphism

## Class
**Кратко:**
Класс — это описание того, какие данные и поведение будут у объектов определенного типа.

**Полное определение:**
Класс задает структуру объекта: поля, свойства, методы, конструкторы, а иногда и ограничения на использование. Он выступает шаблоном, по которому создаются экземпляры. В зависимости от языка класс может также задавать модификаторы доступа, виртуальные методы, статические члены и другую инфраструктуру типа.

**Назначение / зачем это нужно:**
Класс нужен, чтобы собирать вместе связанные данные и операции над ними, а также повторно создавать объекты одной формы.

**Когда применять:**
Когда есть устойчивое понятие предметной области или техническая сущность, для которой полезно явно определить состояние и поведение.

**Ограничения / частые ошибки:**
Ошибка — превращать класс просто в контейнер для случайных методов. Еще одна ошибка — создавать классы ради формальности, когда достаточно простой функции или структуры данных.

**Примеры:**
- Класс `User` описывает имя, email, статус и поведение, связанное с пользователем.
- Класс `Money` может инкапсулировать сумму, валюту и операции сложения или сравнения.

**Связанные термины:**
Object, Encapsulation, Abstraction, Generics

## Object
**Кратко:**
Объект — это конкретный экземпляр класса со своим состоянием в памяти.

**Полное определение:**
Если класс — это описание типа, то объект — его фактическое воплощение во время выполнения. У разных объектов одного класса значения полей могут отличаться, хотя набор полей и методов у них общий. В языках с ссылочной моделью несколько переменных могут ссылаться на один и тот же объект.

**Назначение / зачем это нужно:**
Объект позволяет моделировать конкретную сущность или технический компонент с индивидуальным состоянием.

**Когда применять:**
Когда важно различать отдельные экземпляры, их жизненный цикл, внутреннее состояние и взаимодействие.

**Ограничения / частые ошибки:**
Ошибка — не различать идентичность объекта и равенство по значению. Например, два разных объекта могут быть равны по данным, но оставаться разными экземплярами.

**Примеры:**
- `currentUser` — конкретный объект класса `User`.
- Два объекта `Order` с одинаковой суммой все равно представляют два разных заказа.

**Связанные термины:**
Class, Encapsulation, Reference Type, Value Object

## Abstraction
**Кратко:**
Абстракция позволяет работать с важными свойствами сущности, не погружаясь в лишние детали реализации.

**Полное определение:**
Абстракция — это отбор существенных характеристик модели и сокрытие второстепенных деталей. В коде она проявляется через интерфейсы, базовые классы, доменные типы и API, которые показывают, что можно делать с объектом, но не раскрывают, как именно это реализовано внутри.

**Назначение / зачем это нужно:**
Она снижает когнитивную нагрузку, делает код понятнее и позволяет менять реализацию без изменения внешнего контракта.

**Когда применять:**
Когда нужно отделить использование от реализации, ограничить знание о внутреннем устройстве или построить устойчивый API.

**Ограничения / частые ошибки:**
Плохая абстракция опаснее ее отсутствия. Если абстракция не отражает реальную модель или сценарии использования, она запутывает код и увеличивает связанность вместо снижения.

**Примеры:**
- Интерфейс `IStorage` скрывает, лежат ли данные в PostgreSQL, S3 или локальной файловой системе.
- Тип `Money` скрывает детали округления и проверки валют.

**Связанные термины:**
Interface, Contract, Encapsulation, Polymorphism

## Encapsulation
**Кратко:**
Инкапсуляция объединяет данные и операции над ними, ограничивая неконтролируемый доступ к внутреннему состоянию.

**Полное определение:**
Смысл инкапсуляции не только в `private`-полях. Главное — объект сам отвечает за корректность своего состояния и не позволяет внешнему коду ломать инварианты напрямую. Вместо того чтобы давать произвольный доступ к полям, объект предоставляет методы или свойства с контролируемой логикой.

**Назначение / зачем это нужно:**
Инкапсуляция защищает инварианты, снижает риск некорректных изменений и делает поведение объекта более предсказуемым.

**Когда применять:**
Практически всегда, когда объект хранит состояние, для которого важны правила изменения.

**Ограничения / частые ошибки:**
Ошибка — считать инкапсуляцией любой класс с `private`-полями, хотя состояние можно косвенно ломать через неудачные сеттеры. Другая ошибка — чрезмерно прятать все подряд и делать API неудобным.

**Примеры:**
- Вместо `account.Balance = -100` используется метод `Withdraw`, который проверяет лимиты.
- `Order` сам решает, можно ли перевести его в статус `Paid`.

**Связанные термины:**
Abstraction, Invariant, Class, Contract

## Inheritance
**Кратко:**
Наследование позволяет одному классу унаследовать и при необходимости изменить поведение другого.

**Полное определение:**
При наследовании производный класс получает поля и методы базового класса и может расширять их или переопределять часть поведения. Это создает отношение `is-a`: производный тип считается разновидностью базового. Наследование может быть полезным, но оно сильно связывает классы и переносит ограничения базового класса в иерархию.

**Назначение / зачем это нужно:**
Позволяет переиспользовать общую реализацию и формировать полиморфные иерархии, когда между типами действительно есть устойчивая связь специализации.

**Когда применять:**
Когда связь между типами естественна и стабильна, а базовый класс задает общие правила и контракт для всех наследников.

**Ограничения / частые ошибки:**
Самая частая ошибка — использовать наследование ради повторного использования кода, хотя между типами нет настоящего отношения специализации. Это приводит к хрупким иерархиям, проблемам с LSP и сложному рефакторингу.

**Примеры:**
- `AdminUser` наследуется от `User`, если действительно остается его подтипом во всех важных сценариях.
- Базовый класс `BackgroundJob` задает общий протокол запуска и логирования, а наследники реализуют конкретную работу.

**Связанные термины:**
Composition, Polymorphism, Subtype, Liskov Substitution Principle, Method Overriding

## Composition
**Кратко:**
Композиция строит сложное поведение из более простых объектов, а не через жесткую иерархию наследования.

**Полное определение:**
В композиции один объект использует другие объекты как части своей реализации. Вместо того чтобы говорить «тип B является типом A», композиция говорит «объект использует объект A для выполнения части работы». Это дает более гибкую структуру, потому что поведение можно менять заменой компонентов, а не перестройкой иерархии.

**Назначение / зачем это нужно:**
Композиция снижает связанность, упрощает тестирование и позволяет комбинировать поведение без раздувания иерархий.

**Когда применять:**
В большинстве случаев, когда задача — собрать поведение из зависимостей, стратегий, сервисов и value objects. На практике часто предпочтительнее наследования.

**Ограничения / частые ошибки:**
Ошибка — считать композицию универсальным решением вообще для всего. Иногда базовый класс действительно отражает устойчивую модель. Но по умолчанию композиция безопаснее, потому что меньше привязывает типы друг к другу.

**Примеры:**
- `OrderService` использует `IPaymentGateway` и `IEmailSender` через поля или конструктор.
- `Car` состоит из `Engine`, `Transmission` и `Wheels`, а не наследуется от них.

**Связанные термины:**
Aggregation, Dependency Injection, Interface, Inheritance

## Aggregation
**Кратко:**
Агрегация — отношение «содержит», при котором части могут существовать независимо от целого.

**Полное определение:**
Aggregation обычно рассматривают как более слабую форму композиции. Целое знает о своих частях и использует их, но не обязательно полностью владеет их жизненным циклом. То есть часть может быть создана вне объекта и продолжить существование после его уничтожения.

**Назначение / зачем это нужно:**
Помогает моделировать более мягкие связи между объектами, когда один объект лишь использует другой как составную часть модели.

**Когда применять:**
Когда объект логически включает другие сущности, но не владеет ими целиком.

**Ограничения / частые ошибки:**
На практике языки программирования редко заставляют строго различать композицию и агрегацию. Поэтому важно не столько слово, сколько смысл владения, жизненного цикла и ответственности.

**Примеры:**
- `Team` содержит список `Developer`, но разработчики существуют независимо от конкретной команды.
- `Report` использует `User`, загруженного из репозитория, но не владеет его жизненным циклом.

**Связанные термины:**
Composition, Object, Lifecycle, Association

## Polymorphism
**Кратко:**
Полиморфизм позволяет обращаться к разным реализациям через общий тип или контракт.

**Полное определение:**
Полиморфизм означает, что код может работать не с конкретной реализацией, а с абстракцией. Например, функция принимает интерфейс `IFormatter`, а реальный объект может быть `JsonFormatter`, `XmlFormatter` или `CsvFormatter`. За счет этого поведение выбирается во время выполнения или компиляции, в зависимости от механизма языка.

**Назначение / зачем это нужно:**
Он уменьшает связанность кода с конкретными реализациями, упрощает расширение и повышает тестируемость.

**Когда применять:**
Когда есть несколько взаимозаменяемых реализаций одной роли или сценария.

**Ограничения / частые ошибки:**
Ошибка — вводить полиморфизм без реальной вариативности, просто ради «красивой архитектуры». Если у тебя всегда одна реализация и нет вероятности второй, абстракция может быть преждевременной.

**Примеры:**
- Один и тот же код отправляет уведомление через `INotificationSender`, а конкретный канал выбирается конфигурацией.
- Тест использует fake-реализацию интерфейса вместо реального внешнего клиента.

**Связанные термины:**
Interface, Abstract Class, Contract, Subtype, Generics

## Interface
**Кратко:**
Интерфейс описывает, что объект умеет делать, не раскрывая, как именно он это делает.

**Полное определение:**
Интерфейс — это контракт поведения. Он определяет доступные методы, свойства или события, но не хранит конкретную реализацию, либо хранит ее в ограниченной форме в некоторых языках. Через интерфейсы удобно отделять использование от реализации и строить полиморфный код.

**Назначение / зачем это нужно:**
Интерфейсы снижают связанность, помогают тестировать код и дают устойчивую точку расширения.

**Когда применять:**
Когда у компонента есть несколько возможных реализаций, когда нужно отделить доменную логику от инфраструктуры или когда важно задать ясный контракт взаимодействия.

**Ограничения / частые ошибки:**
Ошибка — создавать интерфейс на каждый класс без причины. Интерфейс полезен там, где есть вариативность, подмена, граница слоя или важный контракт, а не как обязательный ритуал.

**Примеры:**
- `IClock` позволяет подменять текущее время в тестах.
- `IRepository<T>` задает операции доступа к данным без привязки к конкретной базе.

**Связанные термины:**
Contract, Abstraction, Polymorphism, Dependency Inversion Principle

## Abstract Class
**Кратко:**
Абстрактный класс задает общий базовый контракт и может содержать готовую реализацию части поведения.

**Полное определение:**
В отличие от интерфейса, abstract class часто хранит состояние, общие методы и защищенные расширяемые точки для наследников. Он полезен, когда нескольким типам нужна не только общая форма, но и общая реализация. При этом экземпляр абстрактного класса напрямую создать обычно нельзя.

**Назначение / зачем это нужно:**
Позволяет вынести повторяющуюся базовую логику и одновременно задать общий шаблон поведения для наследников.

**Когда применять:**
Когда у семейства типов действительно есть общий алгоритм, состояние или протокол работы, а не только общий набор методов.

**Ограничения / частые ошибки:**
Ошибка — использовать абстрактный класс там, где достаточно интерфейса и композиции. Чем больше логики в базовом классе, тем сильнее связаны наследники с его внутренними решениями.

**Примеры:**
- Базовый `CommandHandler` реализует общую обвязку логирования и валидации, а конкретные обработчики переопределяют только основную операцию.
- `Stream` в некоторых экосистемах задает общий контракт и набор базовых операций.

**Связанные термины:**
Interface, Inheritance, Method Overriding, Template Method

## Contract
**Кратко:**
Контракт — это договоренность о том, как компонент должен себя вести и что от него ожидают потребители.

**Полное определение:**
Контракт включает не только сигнатуры методов, но и семантику: допустимые входы, обещанные выходы, инварианты, побочные эффекты, гарантии ошибок и ограничения. В одних случаях контракт закреплен типовой системой, в других — документацией, тестами или соглашениями команды.

**Назначение / зачем это нужно:**
Контракт делает поведение компонента предсказуемым и уменьшает риск неправильного использования.

**Когда применять:**
Всегда, когда проектируешь публичный API, интерфейс между слоями, библиотеку или важную доменную сущность.

**Ограничения / частые ошибки:**
Ошибка — думать, что контракт исчерпывается названием метода. Без явной семантики `Save`, `Get` или `Validate` могут означать очень разные вещи.

**Примеры:**
- Метод `Withdraw(amount)` обязан отклонять отрицательные суммы и не нарушать инвариант баланса.
- Интерфейс репозитория обещает возвращать доменную модель, а не инфраструктурный DTO.

**Связанные термины:**
Interface, Encapsulation, Invariant, Precondition, Postcondition

## Subtype
**Кратко:**
Подтип — это тип, который может использоваться там, где ожидается другой, более общий тип.

**Полное определение:**
Понятие subtype связано не только с синтаксисом наследования. В широком смысле подтип — это тип, сохраняющий совместимость с ожиданиями базового контракта. В одних языках подтип возникает через наследование, в других — через интерфейсы или структурную совместимость.

**Назначение / зачем это нужно:**
Подтипы лежат в основе полиморфизма и позволяют заменять конкретные реализации без изменения клиентского кода.

**Когда применять:**
Когда модель действительно требует отношения «частный случай общего типа» или когда реализация удовлетворяет общему контракту.

**Ограничения / частые ошибки:**
Ошибка — считать любой наследник корректным подтипом. Формальное наследование еще не гарантирует, что подмена будет безопасной.

**Примеры:**
- `Square` формально может наследоваться от `Rectangle`, но часто нарушает ожидания по изменению ширины и высоты.
- `CachedUserRepository` может быть подтипом `IUserRepository`, если полностью соблюдает его контракт.

**Связанные термины:**
Polymorphism, Inheritance, Interface, Liskov Substitution Principle

## Liskov Substitution Principle
**Кратко:**
LSP требует, чтобы объект подтипа можно было подставить вместо объекта базового типа без нарушения корректности программы.

**Полное определение:**
Принцип Барбары Лисков говорит, что подтип обязан сохранять ожидания базового контракта. Это означает не только совпадение сигнатур, но и совместимость поведения: нельзя ужесточать предусловия, ослаблять постусловия, ломать инварианты или неожиданно менять смысл операций.

**Назначение / зачем это нужно:**
LSP помогает оценивать качество иерархий наследования и предотвращать поломки полиморфного кода.

**Когда применять:**
При проектировании базовых классов, интерфейсов, заменяемых реализаций и доменных иерархий.

**Ограничения / частые ошибки:**
Классическая ошибка — моделировать связь наследованием только по внешнему сходству. Если наследник должен отключать часть базового поведения, бросать `NotSupportedException` или менять семантику метода, иерархия, вероятно, неверна.

**Примеры:**
- `ReadOnlyCollection` не должна наследоваться от абстракции, чей основной контракт подразумевает изменение данных.
- Если `Bird` имеет метод `Fly`, то `Penguin : Bird` может оказаться плохой моделью.

**Связанные термины:**
Subtype, Inheritance, Contract, Polymorphism

## Method Overriding
**Кратко:**
Переопределение метода позволяет наследнику изменить или уточнить реализацию, заданную в базовом классе.

**Полное определение:**
Overriding работает в иерархии наследования: базовый класс объявляет виртуальный, абстрактный или аналогично расширяемый метод, а производный класс дает свою реализацию. При вызове через ссылку базового типа фактическое поведение определяется реальным типом объекта.

**Назначение / зачем это нужно:**
Нужно для полиморфного изменения поведения в рамках общего контракта.

**Когда применять:**
Когда в иерархии есть стабильный общий протокол, но отдельные шаги алгоритма должны отличаться у наследников.

**Ограничения / частые ошибки:**
Ошибка — переопределять метод, нарушая ожидания базового класса. Еще одна ошибка — слишком глубоко строить иерархию, где поведение трудно отследить.

**Примеры:**
- `ToString()` переопределяется для человекочитаемого вывода объекта.
- Базовый `Validate()` задает контракт, а наследники реализуют конкретные проверки.

**Связанные термины:**
Inheritance, Abstract Class, Polymorphism, Liskov Substitution Principle

## Method Overloading
**Кратко:**
Перегрузка метода — это несколько методов с одним именем, но разными параметрами.

**Полное определение:**
Overloading — механизм удобства API, при котором одно и то же имя используется для схожих операций с разными сигнатурами. Выбор конкретного метода обычно происходит на этапе компиляции по набору аргументов.

**Назначение / зачем это нужно:**
Позволяет делать API компактнее и выразительнее, когда операции концептуально одинаковые, но принимают разные входы.

**Когда применять:**
Когда варианты метода действительно описывают одно и то же действие, а различается только форма входных данных.

**Ограничения / частые ошибки:**
Ошибка — перегружать методы слишком агрессивно, так что становится непонятно, какой вариант будет вызван и чем они реально отличаются. Это особенно опасно при optional parameters и неявных преобразованиях.

**Примеры:**
- `Parse(string value)` и `Parse(ReadOnlySpan<char> value)`.
- `Create(UserDto dto)` и `Create(string name, string email)`.

**Связанные термины:**
Method Overriding, API Design, Static Typing

## Generics
**Кратко:**
Обобщения позволяют писать код один раз для множества типов, сохраняя проверки типов на этапе компиляции.

**Полное определение:**
Generics вводят параметры типов: например, `List<T>` или `Repository<T>`. Вместо дублирования логики под `User`, `Order` и `Invoice` можно описать общий алгоритм или структуру данных и подставлять конкретный тип при использовании. Это дает переиспользование без потери типобезопасности и без приведения к базовому `object`.

**Назначение / зачем это нужно:**
Обобщения уменьшают дублирование, делают API выразительными и защищают от ошибок типов.

**Когда применять:**
Когда поведение одинаково для семейства типов или когда нужна универсальная структура данных, алгоритм или сервис.

**Ограничения / частые ошибки:**
Ошибка — делать generic там, где нет реальной общности и параметр типа лишь усложняет понимание. Еще одна ошибка — слишком расплывчатые ограничения на `T`, из-за чего контракт generic-компонента неясен.

**Примеры:**
- `List<T>` работает с любым типом элементов.
- `IRepository<TAggregate>` можно использовать для разных доменных агрегатов при четком общем контракте.

**Связанные термины:**
Class, Type Safety, Interface, Polymorphism
