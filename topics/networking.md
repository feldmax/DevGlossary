## Краткий список терминов
- **Network Protocol** — набор правил, по которым участники сети обмениваются данными.
- **Client-Server** — модель взаимодействия, где клиент запрашивает услугу, а сервер ее предоставляет.
- **TCP** — транспортный протокол с установлением соединения и гарантией доставки данных.
- **UDP** — транспортный протокол без установления соединения и без гарантии доставки.
- **HTTP** — прикладной протокол обмена запросами и ответами, широко используемый в вебе и API.
- **HTTPS** — HTTP поверх TLS, обеспечивающий шифрование и защиту канала.
- **API** — контракт взаимодействия между программными компонентами.
- **REST** — архитектурный стиль построения HTTP API вокруг ресурсов и стандартных методов.
- **RPC** — модель удаленного вызова процедуры, где удаленная операция выглядит как локальный вызов.
- **gRPC** — современный RPC-фреймворк поверх HTTP/2 и Protocol Buffers.
- **WebSocket** — постоянное двустороннее соединение между клиентом и сервером.
- **Statelessness** — подход, при котором сервер не хранит состояние клиента между запросами в сессии взаимодействия.
- **Idempotency** — свойство операции, при котором повторный вызов дает тот же эффект, что и один вызов.
- **Timeout** — ограничение времени ожидания ответа или завершения операции.
- **Retry** — повтор неудачной операции по заданным правилам.
- **Rate Limiting** — ограничение частоты запросов к сервису.
- **Load Balancing** — распределение нагрузки между несколькими экземплярами сервиса.
- **Consistency** — свойство согласованности данных между участниками системы.
- **Eventual Consistency** — модель, при которой данные становятся согласованными не сразу, а со временем.
- **CAP Theorem** — формулировка о компромиссе между consistency, availability и partition tolerance в распределенной системе.

## Network Protocol
**Кратко:**
Network protocol — это формальный набор правил, который определяет формат сообщений, порядок обмена и ожидаемое поведение участников сети.

**Полное определение:**
Протокол задает, как именно устройства или процессы должны передавать данные: как устроен пакет, как начинается и завершается соединение, как подтверждается получение, как обрабатываются ошибки и какие действия считаются корректными. Без протокола два участника не смогут интерпретировать сообщения одинаково.

**Назначение / зачем это нужно:**
Протокол нужен для совместимости и предсказуемого обмена данными между разными программами, машинами и сетевыми слоями.

**Когда применять:**
Всегда, когда есть взаимодействие по сети: от низкоуровневой передачи пакетов до прикладных API между сервисами.

**Ограничения / частые ошибки:**
Ошибка — думать, что протокол описывает только формат данных. На практике он задает еще и семантику, последовательность действий, таймауты и сценарии ошибок.

**Примеры:**
- HTTP описывает формат запроса, ответа, заголовков и статус-кодов.
- TCP определяет порядок доставки, подтверждения и повторной отправки сегментов.

**Связанные термины:**
TCP, UDP, HTTP, API, Handshake

## Client-Server
**Кратко:**
Client-server — это модель, в которой один участник инициирует запрос, а другой обрабатывает его и возвращает результат.

**Полное определение:**
В клиент-серверной модели роли разделены: клиент отвечает за инициирование взаимодействия и потребление результата, сервер — за предоставление данных или услуг. Один сервер может обслуживать множество клиентов, а один клиент — взаимодействовать с несколькими серверами. Эта модель лежит в основе веба, мобильных приложений и большинства backend-систем.

**Назначение / зачем это нужно:**
Модель нужна для разделения обязанностей, централизованного управления данными и масштабирования обслуживания пользователей.

**Когда применять:**
Когда есть пользовательский интерфейс, внешняя система или другой сервис, которому нужны данные или операции от выделенного сервера.

**Ограничения / частые ошибки:**
Ошибка — воспринимать клиент и сервер как всегда разные машины. Это логические роли: они могут находиться и на одном хосте.

**Примеры:**
- Браузер отправляет запрос сайту, а сервер возвращает HTML.
- Frontend-приложение запрашивает данные у backend API.

**Связанные термины:**
HTTP, API, Request, Response, Load Balancing

## TCP
**Кратко:**
TCP — это транспортный протокол, который обеспечивает надежную, упорядоченную и контролируемую доставку потока данных.

**Полное определение:**
TCP устанавливает соединение между двумя участниками, разбивает данные на сегменты, отслеживает подтверждения получения, переотправляет потерянные части и гарантирует правильный порядок доставки. Он ориентирован на надежность, а не на минимальную задержку любой ценой.

**Назначение / зачем это нужно:**
TCP нужен там, где важно получить все данные без потерь и в правильной последовательности.

**Когда применять:**
Для веба, API, баз данных, передачи файлов, почтовых протоколов и большинства прикладных систем.

**Ограничения / частые ошибки:**
Ошибка — считать TCP «быстрее по умолчанию». За надежность он платит накладными расходами, установлением соединения и контролем перегрузки.

**Примеры:**
- HTTP/1.1 и HTTP/2 обычно работают поверх TCP.
- Подключение к PostgreSQL использует TCP-соединение.

**Связанные термины:**
UDP, HTTP, Connection, Packet Loss, Retransmission

## UDP
**Кратко:**
UDP — это транспортный протокол без установления соединения, который отправляет датаграммы без гарантии доставки и порядка.

**Полное определение:**
UDP минималистичен: он позволяет быстро отправлять независимые сообщения без подтверждений, переотправок и встроенного контроля порядка. Это снижает накладные расходы и задержку, но перекладывает вопросы надежности на приложение, если они нужны.

**Назначение / зачем это нужно:**
UDP нужен для сценариев, где важнее скорость, простота и низкая задержка, чем стопроцентная надежность доставки каждого пакета.

**Когда применять:**
Для стриминга, голосовой связи, телеметрии, игр в реальном времени, DNS и других latency-sensitive сценариев.

**Ограничения / частые ошибки:**
Ошибка — использовать UDP как замену TCP, не продумав, что делать с потерями, дубликатами и нарушением порядка сообщений.

**Примеры:**
- DNS-запросы часто идут по UDP.
- В онлайн-играх координаты игрока могут отправляться по UDP.

**Связанные термины:**
TCP, Datagram, Packet Loss, Real-Time Systems

## HTTP
**Кратко:**
HTTP — это прикладной протокол для обмена запросами и ответами между клиентом и сервером.

**Полное определение:**
HTTP задает, как клиент формирует запрос, какие методы использует, как передаются заголовки, тело и статус ответа. Он не описывает бизнес-логику приложения, но задает транспортный контракт взаимодействия на прикладном уровне. Это фундамент веб-сайтов, REST API и множества интеграций.

**Назначение / зачем это нужно:**
HTTP нужен как универсальный и широко поддерживаемый способ удаленного взаимодействия между системами.

**Когда применять:**
Для веб-приложений, публичных и внутренних API, интеграций с браузерами, мобильными приложениями и внешними сервисами.

**Ограничения / частые ошибки:**
Ошибка — смешивать HTTP как протокол транспорта с REST как архитектурным стилем. HTTP может использоваться и вне REST.

**Примеры:**
- `GET /users/42` получает ресурс по HTTP.
- Сервис возвращает код `404 Not Found`, если объект не найден.

**Связанные термины:**
HTTPS, REST, API, Status Code, Header

## HTTPS
**Кратко:**
HTTPS — это HTTP, защищенный криптографическим слоем TLS.

**Полное определение:**
HTTPS добавляет шифрование канала, проверку подлинности сервера через сертификаты и защиту от подмены и пассивного перехвата данных. Для клиента взаимодействие похоже на обычный HTTP, но транспорт становится безопасным для передачи чувствительных данных.

**Назначение / зачем это нужно:**
HTTPS нужен для конфиденциальности, целостности данных и доверия к серверу.

**Когда применять:**
Практически всегда во внешних и во многих внутренних сетевых взаимодействиях, особенно при передаче логинов, токенов, персональных данных и бизнес-критичных запросов.

**Ограничения / частые ошибки:**
Ошибка — думать, что HTTPS автоматически делает всю систему безопасной. Он защищает канал, но не исправляет уязвимости приложения.

**Примеры:**
- Авторизация пользователя должна идти по HTTPS.
- Внешнее API публикуется только по TLS-защищенному адресу.

**Связанные термины:**
HTTP, TLS, Certificate, Authentication, Security

## API
**Кратко:**
API — это контракт, через который один программный компонент предоставляет возможности другому.

**Полное определение:**
API определяет доступные операции, формат входных и выходных данных, ограничения, ошибки, правила авторизации и семантику вызовов. API может быть сетевым, библиотечным, файловым или системным. В контексте backend-разработки под API обычно понимают сетевой интерфейс сервиса.

**Назначение / зачем это нужно:**
API нужен для формального и повторяемого взаимодействия между компонентами, командами и системами.

**Когда применять:**
Когда один модуль, сервис, библиотека или внешняя система должен предоставлять другому предсказуемый интерфейс.

**Ограничения / частые ошибки:**
Ошибка — воспринимать API только как набор URL. Полноценный API включает модель данных, семантику, версионирование, ошибки и контракт совместимости.

**Примеры:**
- Backend предоставляет HTTP API для мобильного клиента.
- Библиотека exposes API в виде публичных классов и функций.

**Связанные термины:**
HTTP, REST, RPC, Contract, Versioning

## REST
**Кратко:**
REST — это архитектурный стиль, в котором API строится вокруг ресурсов, стандартных HTTP-методов и единообразного интерфейса.

**Полное определение:**
REST не является отдельным протоколом. Это набор архитектурных ограничений для проектирования API: ресурсы идентифицируются URI, операции выражаются стандартными методами HTTP, взаимодействие stateless, а представления ресурсов передаются в ответах. На практике «REST API» часто означает просто «HTTP API», хотя строгое соответствие REST встречается не всегда.

**Назначение / зачем это нужно:**
REST нужен для предсказуемой и понятной организации HTTP API с хорошей совместимостью с экосистемой веба.

**Когда применять:**
Когда строится публичное или внутреннее HTTP API с ресурсной моделью, CRUD-операциями и стандартным поведением методов.

**Ограничения / частые ошибки:**
Ошибка — называть REST любой HTTP endpoint. Если API игнорирует ресурсную модель, методы и stateless-подход, это может быть просто HTTP API, а не REST в строгом смысле.

**Примеры:**
- `GET /orders/42` читает заказ.
- `POST /orders` создает новый заказ.

**Связанные термины:**
HTTP, API, Statelessness, Resource, Idempotency

## RPC
**Кратко:**
RPC — это модель взаимодействия, при которой удаленный вызов выглядит как вызов функции или метода.

**Полное определение:**
В RPC клиент обращается к удаленной операции по имени или контракту, передает аргументы и получает результат, как будто вызывает локальную функцию. Под капотом есть сериализация, сеть, обработка ошибок, таймауты и преобразование данных. RPC скрывает транспортные детали, но не отменяет сетевую природу вызова.

**Назначение / зачем это нужно:**
RPC нужен для удобного вызова удаленной логики с четким контрактом и процедурной моделью.

**Когда применять:**
Когда взаимодействие естественно моделируется как вызов операций, а не как работа с HTTP-ресурсами.

**Ограничения / частые ошибки:**
Ошибка — забывать, что удаленный вызов не равен локальному: сеть ненадежна, задержка выше, возможны частичные отказы и повторные вызовы.

**Примеры:**
- Сервис вызывает `CreateInvoice` на удаленном billing-сервисе.
- Внутренние микросервисы общаются через RPC-контракты.

**Связанные термины:**
API, gRPC, Timeout, Retry, Serialization

## gRPC
**Кратко:**
gRPC — это RPC-фреймворк, который использует HTTP/2 и Protocol Buffers для описания и передачи сообщений.

**Полное определение:**
gRPC формализует контракты через `.proto`-схемы, генерирует клиентский и серверный код, поддерживает двоичную сериализацию, стриминг и эффективное взаимодействие между сервисами. Он особенно удобен для внутренней service-to-service коммуникации, где важны производительность и строгие контракты.

**Назначение / зачем это нужно:**
gRPC нужен для типизированных, быстрых и хорошо формализованных межсервисных вызовов.

**Когда применять:**
Во внутренних распределенных системах, когда обе стороны контролируются одной организацией и готовы работать с protobuf и кодогенерацией.

**Ограничения / частые ошибки:**
Ошибка — выбирать gRPC для публичного браузерного API по привычке. Для браузеров и внешних интеграций HTTP/JSON часто проще.

**Примеры:**
- Сервис пользователей и сервис заказов общаются по gRPC.
- `.proto` описывает сообщения `UserRequest` и `UserResponse`.

**Связанные термины:**
RPC, API, HTTP/2, Protocol Buffers, Streaming

## WebSocket
**Кратко:**
WebSocket — это протокол постоянного двустороннего соединения, позволяющий обеим сторонам отправлять сообщения в любой момент.

**Полное определение:**
После начального handshake WebSocket превращает HTTP-соединение в постоянный канал обмена сообщениями. В отличие от обычного request-response паттерна, сервер может сам инициировать отправку данных клиенту. Это удобно для real-time сценариев.

**Назначение / зачем это нужно:**
WebSocket нужен для событийного и двустороннего взаимодействия с низкой задержкой без постоянного polling.

**Когда применять:**
Для чатов, торговых терминалов, игровых сессий, live dashboards, уведомлений и иных real-time систем.

**Ограничения / частые ошибки:**
Ошибка — использовать WebSocket просто потому, что он «современнее». Если достаточно обычных запросов и периодического обновления, HTTP может быть проще и дешевле в эксплуатации.

**Примеры:**
- Сервер отправляет клиенту новое сообщение чата сразу после получения.
- Dashboard получает обновления метрик в реальном времени.

**Связанные термины:**
HTTP, Event Loop, Push, Streaming, Real-Time Systems

## Statelessness
**Кратко:**
Statelessness — это подход, при котором каждый запрос самодостаточен и не требует, чтобы сервер помнил контекст предыдущих запросов клиента.

**Полное определение:**
В stateless-взаимодействии все необходимые данные для обработки запроса передаются вместе с самим запросом: идентификаторы, токены, параметры, версия ресурса и так далее. Сервер не опирается на локальное состояние конкретной сессии. Это упрощает масштабирование и балансировку нагрузки.

**Назначение / зачем это нужно:**
Statelessness нужен для горизонтального масштабирования, предсказуемости и снижения связности между запросами.

**Когда применять:**
В большинстве HTTP API и публичных backend-сервисов, где важно свободно распределять запросы между экземплярами.

**Ограничения / частые ошибки:**
Ошибка — путать stateless с отсутствием состояния вообще. Состояние системы может храниться в базе, кеше или токене, просто не в памяти конкретного процесса между запросами.

**Примеры:**
- Каждый запрос содержит JWT, а сервер не хранит server-side session.
- Любой экземпляр сервиса может обработать любой запрос пользователя.

**Связанные термины:**
REST, Session, Load Balancing, Horizontal Scaling

## Idempotency
**Кратко:**
Idempotency — это свойство операции, при котором повторное выполнение с теми же входными данными не меняет итоговый эффект по сравнению с одним выполнением.

**Полное определение:**
Идемпотентная операция может быть вызвана один раз или много раз, но система после завершения окажется в том же состоянии, что и после первого успешного вызова. Это особенно важно в сетевых системах, где из-за сбоев, таймаутов или повторной отправки клиент не всегда знает, был ли предыдущий запрос обработан.

**Назначение / зачем это нужно:**
Идемпотентность нужна для безопасных повторов, устойчивости к дублям и корректной работы распределенных и ненадежных коммуникаций.

**Когда применять:**
Для операций создания, списания, обновления и команд, которые могут быть отправлены повторно из-за retry-механизмов или проблем сети.

**Ограничения / частые ошибки:**
Ошибка — путать идемпотентность с тем, что операция «ничего не делает». Она может менять состояние, но повторный вызов не должен менять его еще раз.

**Примеры:**
- `PUT /users/42` с одинаковым телом можно безопасно повторить.
- Платежный API использует idempotency key, чтобы двойной retry не создал два списания.

**Связанные термины:**
REST, Retry, Timeout, Duplicate Delivery, Exactly-Once

---

## Timeout
**Кратко:**
Timeout — это ограничение максимального времени, которое система готова ждать операцию или ответ.

**Полное определение:**
Таймаут завершает ожидание, если ответ не пришел вовремя. Это защищает поток выполнения, соединения, ресурсы и user experience от бесконционного зависания. Таймауты бывают на установление соединения, чтение, общий запрос, отдельный шаг и так далее.

**Назначение / зачем это нужно:**
Timeout нужен для управления отказами, освобождения ресурсов и предсказуемого поведения системы под проблемами сети и зависшими зависимостями.

**Когда применять:**
Практически для любых внешних вызовов: HTTP, RPC, база данных, очередь, файловое хранилище.

**Ограничения / частые ошибки:**
Ошибка — ставить таймауты слишком большими или вообще не ставить. Другая ошибка — задавать слишком агрессивные значения без учета нормальной латентности.

**Примеры:**
- HTTP-клиент ждет не более 3 секунд.
- Подключение к БД прерывается, если за 5 секунд соединение не установлено.

**Связанные термины:**
Retry, Circuit Breaker, Latency, Failure Mode

---

## Retry
**Кратко:**
Retry — это повтор операции после временной неудачи в надежде, что следующая попытка пройдет успешно.

**Полное определение:**
Повторы применяются к transient failures: кратковременным сетевым сбоям, временной недоступности узла, перегрузке или истечению таймаута. Корректный retry обычно включает лимит попыток, задержку между ними и часто — backoff и jitter. Без этого ретраи могут усилить перегрузку системы.

**Назначение / зачем это нужно:**
Retry нужен для повышения устойчивости к временным отказам в распределенных системах.

**Когда применять:**
Когда ошибка потенциально кратковременная и повтор имеет смысл, а сама операция безопасна для повтора или идемпотентна.

**Ограничения / частые ошибки:**
Ошибка — ретраить любые ошибки подряд, включая валидационные и бизнес-ошибки. Еще одна ошибка — повторять неидемпотентную операцию без защиты от дублей.

**Примеры:**
- Клиент повторяет запрос к сервису через exponential backoff.
- Воркеры повторяют публикацию события после временного сетевого сбоя.

**Связанные термины:**
Timeout, Idempotency, Backoff, Transient Failure

## Rate Limiting
**Кратко:**
Rate limiting — это ограничение количества запросов или операций за единицу времени.

**Полное определение:**
Ограничение скорости защищает систему от перегрузки, злоупотреблений и неравномерного потребления ресурсов. Лимиты могут применяться по пользователю, токену, IP, tenant, endpoint или глобально. Реализации часто используют token bucket, leaky bucket или fixed/sliding window.

**Назначение / зачем это нужно:**
Rate limiting нужен для стабильности системы, fair usage и защиты от всплесков нагрузки и abuse.

**Когда применять:**
Во внешних API, публичных сервисах, дорогих операциях и любой системе с ограниченными вычислительными ресурсами.

**Ограничения / частые ошибки:**
Ошибка — вводить лимиты без понятной модели ключа и без прозрачной обратной связи клиенту. Пользователь должен понимать, когда и почему его ограничили.

**Примеры:**
- API разрешает 100 запросов в минуту на API key.
- Endpoint генерации отчетов ограничен до 5 запусков в час.

**Связанные термины:**
Backpressure, Throttling, Load Balancing, Availability

## Load Balancing
**Кратко:**
Load balancing — это распределение входящей нагрузки между несколькими узлами или экземплярами сервиса.

**Полное определение:**
Балансировщик принимает запросы и направляет их на один из доступных backend-экземпляров по определенной стратегии: round robin, least connections, weighted routing и так далее. Это улучшает масштабируемость, отказоустойчивость и утилизацию ресурсов.

**Назначение / зачем это нужно:**
Load balancing нужен, чтобы ни один экземпляр не стал узким местом и чтобы система могла переживать рост нагрузки и отказ отдельных узлов.

**Когда применять:**
Когда сервис развернут в нескольких экземплярах или когда нужно обеспечить failover и горизонтальное масштабирование.

**Ограничения / частые ошибки:**
Ошибка — забывать про sticky sessions, health checks, statelessness и реальные ограничения downstream-компонентов вроде базы данных.

**Примеры:**
- Nginx распределяет запросы между несколькими приложениями.
- Kubernetes Service балансирует трафик между pod'ами.

**Связанные термины:**
Statelessness, Horizontal Scaling, Health Check, Availability

## Consistency
**Кратко:**
Consistency — это свойство системы, при котором разные участники видят данные в согласованном и логически корректном состоянии.

**Полное определение:**
В распределенных системах согласованность означает, насколько быстро и насколько одинаково все реплики и клиенты видят обновления. В более общем смысле это предсказуемость модели данных: после операции чтение дает ожидаемый результат без логических противоречий. Разные системы предлагают разные уровни consistency.

**Назначение / зачем это нужно:**
Consistency нужна для корректной бизнес-логики, когда несколько узлов и клиентов работают с одними и теми же данными.

**Когда применять:**
Всегда, когда данные реплицируются, кэшируются или читаются и пишутся несколькими участниками одновременно.

**Ограничения / частые ошибки:**
Ошибка — использовать слово consistency слишком абстрактно. Нужно уточнять, о какой именно модели идет речь: strong, eventual, read-your-writes и так далее.

**Примеры:**
- После изменения профиля пользователь сразу видит новую версию на всех узлах.
- В финансовой системе остаток не должен отличаться между репликами в критической операции.

**Связанные термины:**
Eventual Consistency, Replication, CAP Theorem, Transaction

---

## Eventual Consistency
**Кратко:**
Eventual consistency — это модель, при которой система допускает временные расхождения данных, но гарантирует, что при отсутствии новых изменений они со временем сойдутся.

**Полное определение:**
Такая модель часто используется в реплицируемых и геораспределенных системах, где мгновенная строгая согласованность слишком дорога или ухудшает доступность. Обновление сначала может быть видно не всем узлам, но позже изменения распространяются и реплики приходят к одному состоянию.

**Назначение / зачем это нужно:**
Eventual consistency нужна, когда масштаб, география или требования к доступности важнее мгновенной глобальной согласованности.

**Когда применять:**
Для кэшей, распределенных хранилищ, репликации между регионами, систем с высокой доступностью и tolerant-бизнес-сценариями.

**Ограничения / частые ошибки:**
Ошибка — использовать eventual consistency там, где бизнес ожидает немедленной точности, например в критичных финансовых подтверждениях без дополнительной защиты.

**Примеры:**
- Обновление профиля пользователя может появиться на всех edge-узлах не мгновенно.
- Реплика базы в другом регионе догоняет primary с задержкой.

**Связанные термины:**
Consistency, Replication, CAP Theorem, Availability

## CAP Theorem
**Кратко:**
CAP theorem — это идея о том, что при сетевом разделении распределенная система не может одновременно гарантировать и полную consistency, и полную availability.

**Полное определение:**
Теорема CAP рассматривает три свойства: consistency, availability и partition tolerance. В реальной распределенной сети возможность partition нужно принимать как данность, поэтому при разделении приходится выбирать приоритет: либо возвращать потенциально несогласованные данные, либо временно отказывать в части операций ради согласованности. Теорема не говорит, что нужно выбрать только два свойства «навсегда»; она описывает компромисс именно в условиях partition.

**Назначение / зачем это нужно:**
CAP нужен как модель мышления при проектировании распределенных систем и выборе поведения при сбоях сети.

**Когда применять:**
Когда система имеет несколько узлов, репликацию, межрегиональное взаимодействие или зависит от сетевой связности между частями кластера.

**Ограничения / частые ошибки:**
Частая ошибка — упрощать CAP до лозунга «выбери любые два из трех». Практически система может вести себя по-разному в нормальном режиме и при сетевом разделении.

**Примеры:**
- При partition система может продолжить отвечать чтением со stale-реплики, жертвуя strict consistency.
- Другой сервис может временно отклонять запись, сохраняя согласованность между узлами.

**Связанные термины:**
Consistency, Eventual Consistency, Availability, Partition Tolerance
